## Active Plan Contract Extract

Description themes taken from `planning/0000055-users-can-queue-ingest-and-re-embed-requests.md`:

- Queueable ingest and re-embed requests must persist in Mongo instead of failing fast when another run is active.
- Queue identity (`requestId`) and runtime identity (`runId`) are intentionally different.
- The queue uses one collection with `waiting`, `running`, and `cleanup-blocked`.
- Waiting duplicates update in place without changing queue identity, queue order, original `createdAt`, or original `sourceSurface`.
- Blocking re-embed callers still block until queued work reaches terminal completion.
- The shared repository-list payload is the source of truth for queued visibility, including temporary queued rows for brand-new roots.
- The ingest UI must keep queueable submission enabled while another run is active, while destructive controls may remain gated.

## Fast-Path And Dependency Ordering Notes

- `pumpIngestQueue()` returns before any provider/client work when:
  - a queue cleanup finalizer is already running
  - another ingest lock owner is active
  - no waiting queue item exists after promotion is attempted
- `recoverIngestQueueOnStartup()` returns before provider/client work when:
  - an active ingest lock already exists
  - there is no recoverable queue work
- Dependency-free invariant to challenge later:
  - no-op queue paths should complete without embedding, network, or provider work, and must not fabricate a `runId` unless a waiting request is actually promoted
