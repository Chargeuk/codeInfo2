## Findings

### 1. `must_fix` `plan_contract_issue` `current_repository`

Post-production cancellation can deadlock the ingest worker if queued embedding work is still waiting when the user cancels.

Evidence:

- [server/src/ingest/embeddingDispatcher.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/embeddingDispatcher.ts#L104) only resolves `waitForIdle()` when both `queue.length === 0` and `controllers.size === 0`.
- [server/src/ingest/embeddingDispatcher.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/embeddingDispatcher.ts#L223) aborts in-flight requests on `cancel()`, but it does not clear queued items or otherwise mark queued work terminal.
- [server/src/ingest/embeddingDispatcher.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/embeddingDispatcher.ts#L179) stops `pump()` as soon as `isCancelled()` is true, so queued items remain stranded after cancel.
- [server/src/ingest/ingestJob.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/ingestJob.ts#L1318) calls `dispatcher.completeProduction()` and then awaits `dispatcher.waitForIdle()` before it performs another cancellation check.

Why this matters:

- If the producer has already finished enqueueing chunks, and the user cancels while there are still queued items behind active requests, `cancelRun()` will abort the active requests but the queued items never drain.
- `processRun()` then waits forever in `waitForIdle()`, so the background worker never reaches its final cleanup path. That breaks the story’s cancellation contract, which requires coherent bounded cancellation even with multiple embedding requests in flight.

Proof strength:

- Missing direct proof for this exact lifecycle. The changed cancel tests in [server/src/test/unit/ingest-cancel.test.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/test/unit/ingest-cancel.test.ts#L334) cover cancel-before-result, not cancel-after-production-with-queued-work.

### 2. `must_fix` `plan_contract_issue` `current_repository`

The late-result fence still allows a cancelled run to persist vectors if cancellation lands after `embedBatch()` returns but before `onCompleted()` finishes writing the ordered results buffer.

Evidence:

- [server/src/ingest/embeddingDispatcher.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/embeddingDispatcher.ts#L142) checks `isCancelled()` only once, immediately after `embedBatch()` resolves.
- [server/src/ingest/embeddingDispatcher.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/embeddingDispatcher.ts#L155) then calls `onCompleted(...)` without a second cancellation fence.
- [server/src/ingest/ingestJob.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/ingestJob.ts#L823) drains `pendingResults` into the Chroma batches, and [server/src/ingest/ingestJob.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/ingestJob.ts#L865) schedules that persistence chain independently of the later cancel cleanup.
- [server/src/ingest/ingestJob.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/ingestJob.ts#L1130) deletes vectors for the cancelled run, but there is no protection against an already-running `queuePersist()` adding new vectors again after that delete.

Why this matters:

- The story plan explicitly says late results must be ignored after cancel. In the current code, a cancel that lands in the narrow window after `embedBatch()` resolves but before `queuePersist()` finishes can still write vectors and metadata for a cancelled run.
- That leaves cancellation correctness dependent on a timing coincidence rather than a hard fence.

Proof strength:

- Missing direct proof for the `cancel-after-result-before-persist` race. The changed unit coverage only proves the easier `cancel-before-result` path in [server/src/test/unit/ingest-cancel.test.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/test/unit/ingest-cancel.test.ts#L334).

### 3. `must_fix` `plan_contract_issue` `current_repository`

The exported cancel route can still trigger a fresh embedding-dimension probe after cancellation if the new collection-dimension lookup falls back to `null`.

Evidence:

- [server/src/ingest/ingestJob.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/ingestJob.ts#L403) introduced `resolveCollectionDimension(...)`, but it silently returns `null` for every non-OK response and every fetch error.
- [server/src/ingest/ingestJob.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/ingestJob.ts#L1777) is the exported `cancelRun(...)` used by the live cancel route in [server/src/routes/ingestCancel.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/routes/ingestCancel.ts#L18).
- [server/src/ingest/ingestJob.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/ingestJob.ts#L1814) falls back to `resolveRootEmbeddingDim(...)` when the known dimension is still `<= 1`.
- [server/src/ingest/ingestJob.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/ingestJob.ts#L385) shows that `resolveRootEmbeddingDim(...)` still performs a fresh `embedText(..., 'dimension probe')` request when it cannot infer dimensions from storage.

Why this matters:

- Story 54’s cancellation contract is “stop sending new embedding work immediately.” The route-level cancel path is still capable of issuing a new provider call after cancellation if the collection-dimension lookup fails or the lock metadata is unavailable.
- That reintroduces the exact class of cancel-path behavior the story notes say had to be fixed.

Proof strength:

- Direct proof exists only for the happy path where an existing dimension can be recovered, in [server/src/test/unit/ingest-cancel.test.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/test/unit/ingest-cancel.test.ts#L300). There is missing proof for the lookup-fails branch.
