# Review Findings: Story 0000053

## Findings

### 1. Legacy parent backfill is validated too late, so a mid-request failure can strand resume state in an unrecoverable mixed shape

- Severity: `must_fix`
- Issue Type: `plan_contract_issue`
- Repository Scope: `current_repository`
- Proof Strength: `direct`
- Files:
  - [server/src/flows/service.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/flows/service.ts#L3602)

`startFlowRun(...)` parses a legacy parent flow with no `flags.flow.executionId`, then immediately calls `validateResumeAgentConversations(resumeState)` before the freshly minted parent `executionId` is persisted. That validator can stamp `flags.flowChild.executionId` onto legacy child conversations via `ensureFlowChildConversationOwnership(...)`, but the parent execution id is only written later through `persistFlowResumeState(...)`.

That ordering violates the story contract that a compatible resume path must mint and persist the missing parent `executionId` before later slot reuse depends on it. If the process fails after a child gets stamped but before the parent flow state is saved, the next resume attempt will mint a different parent execution id in memory and then reject the already-stamped child as a conflicting execution marker. The result is a legacy stopped flow that becomes unrecoverable only because the first backfill attempt failed mid-flight.

This path only has happy-path proof today. The current tests cover successful legacy backfill and conflicting child markers, but they do not cover the mixed-shape partial-failure window created by validating and mutating child state before the parent backfill is durable.

### 2. The new Flows page proof asserts the opposite of the planned custom-title contract for fresh runs from an older selected conversation

- Severity: `should_fix`
- Issue Type: `plan_contract_issue`
- Repository Scope: `current_repository`
- Proof Strength: `direct`
- Files:
  - [client/src/test/flowsPage.run.test.tsx](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/test/flowsPage.run.test.tsx#L622)
  - [client/src/pages/FlowsPage.tsx](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/pages/FlowsPage.tsx#L997)
  - [client/src/api/flows.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/api/flows.ts#L141)

The story plan says to preserve the existing custom-title rule for brand-new runs. The production code still does that: a `Run` click always sets `isNewConversation = true`, and the request helper includes `customTitle` whenever the run is new and the trimmed title is non-empty.

The changed test `starts a fresh run with a new conversation id even when an older flow conversation is selected` now asserts `expect(body.customTitle).toBeUndefined()`. That is the opposite of the planned behavior and the opposite of what the current production payload code is written to do. Even if this test currently passes because of incidental UI-state behavior, it is not valid acceptance proof for Story 53 and would normalize the wrong contract in later refactors.
