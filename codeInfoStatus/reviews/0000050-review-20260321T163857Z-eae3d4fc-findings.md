- `title`: Portable selector lookup drops operational failures and re-labels them as invalid user input

In [server/src/ingest/reingestExecution.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/reingestExecution.ts#L191), `canonicalizeSelector()` catches every exception from `resolveRepositorySelector()` and silently falls back to the raw `sourceId`. The caller then feeds that raw selector into the strict single-repo service at [server/src/ingest/reingestExecution.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/reingestExecution.ts#L253), which only understands canonical absolute repository paths. A transient `listIngestedRepositories()` failure therefore turns a valid portable selector into an `INVALID_SOURCE_ID`/`non_absolute` style validation error instead of surfacing the real operational failure.

This is not just a test gap. The same function lets `target: "current"` and `target: "all"` propagate repository-listing failures directly at [server/src/ingest/reingestExecution.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/reingestExecution.ts#L284), so the changed code now reports different failure classes for the same underlying outage depending only on target mode. That drops diagnostics from the selector helper path and violates the story’s expectation that pre-start failures stay clear and actionable.

- Proof status: `missing`
- Why this matters: a portable-selector outage is currently indistinguishable from bad user input, which will send operators toward the wrong fix and makes the error vocabulary inconsistent across `sourceId` versus `current/all` execution paths.

## Finding 2

- `severity`: `should_fix`
- `issue_type`: `generic_engineering_issue`
- `affected_repos`: `[ "codeInfo2" ]`
- `title`: Host-port preflight can silently miss occupied ports when the docker-host probe image cannot run

In [scripts/docker-compose-with-env.sh](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/scripts/docker-compose-with-env.sh#L417), the `auto` branch of `is_port_occupied()` runs a docker-host probe with `docker run --network host --entrypoint node "${probe_image}" ...` whenever the wrapper itself is inside a container. The function then returns that command’s exit code directly at [scripts/docker-compose-with-env.sh](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/scripts/docker-compose-with-env.sh#L432). If the probe image is missing, not yet built, or otherwise cannot start, the non-zero exit code is treated by the caller as “port not occupied,” and the function never falls back to any secondary check.

That means the wrapper can claim the checked-in host-network ports are free when the probe mechanism itself failed. The happy-path shell test at [scripts/test/bats/docker-compose-with-env.bats](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/scripts/test/bats/docker-compose-with-env.bats#L116) proves only that docker-host probing works when the probe image launches successfully; there is no negative coverage for the image-missing or probe-failed case.

- Proof status: `indirect`
- Why this matters: the story explicitly relies on wrapper-first fail-fast host-port validation. A broken probe path that degrades into an implicit pass weakens that contract exactly in the containerized environments where the docker-host check is supposed to keep the result honest.

## Finding 3

- `issue_type`: `plan_contract_issue`
- `affected_repos`: `[ "codeInfo2" ]`
- `title`: Final Story 50 docs still tell readers to use `CODEINFO_MCP_PORT` even though the story removed that legacy env contract

The Task 16 documentation update did not fully complete the env-name migration. [docs/developer-reference.md](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/docs/developer-reference.md#L131) still says the second JSON-RPC server is on `CODEINFO_MCP_PORT`, [docs/developer-reference.md](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/docs/developer-reference.md#L153) still lists `CODEINFO_MCP_PORT` as a live reader, and [design.md](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/design.md#L5145) still describes MCP v2 as listening on `CODEINFO_MCP_PORT`. Those statements conflict with the active story contract, which says the final end state removes the temporary legacy fallback and keeps `CODEINFO_CHAT_MCP_PORT` as the canonical dedicated MCP env name, and they also conflict with the Task 15/16 proof language that `DEV-0000050:T07:checked_in_mcp_contract_loaded` shows `legacyFallbackUsed: false`.

This is more than wording drift: these docs now instruct operators to set an env name the code no longer treats as canonical. Because the story explicitly migrated the checked-in env/docs contract, this should not be left for a later cleanup.

- Proof status: `direct`
- Why this matters: the docs currently describe an obsolete configuration surface for exactly the area Story 50 was meant to finalize, so a reader following the documentation can end up with a misleading setup.
