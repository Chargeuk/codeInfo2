### Description

Users can currently start ingest and re-embed work only when the server is idle. If another ingest operation is already running, later requests are rejected as busy instead of being remembered and run in order. This is frustrating for both humans and AI automation because the user must retry later, and a server restart loses the intent to run that work entirely.

This story adds a durable whole-request queue for ingest and re-embed work. The queue is intentionally about request orchestration, not about how one ingest run chunks files or dispatches embeddings internally. When a request arrives, the server should normalize it to the canonical embed target path, store one queue record in MongoDB, and either start it immediately if nothing is active or leave it waiting until earlier queued work finishes.

Queue-aware success responses now need two identifiers with different jobs. `requestId` is the durable queue identifier for the queued request itself. `runId` remains the runtime ingest identifier once work has actually started. A waiting item can therefore return `requestId` without an active `runId` yet, while an immediately started or later-started item can expose both.

The queue should use one MongoDB collection for both waiting and currently running items. The currently running item stays in the collection until it reaches a terminal outcome. Once that run completes, the server must remove that queue record before it attempts to start the next oldest item. If that delete fails, the server should log an error, surface an error state in the frontend, and keep retrying the delete with backoff while the queue remains stalled. This ordering matters because a restart should be able to look at the same collection and simply start the oldest remaining item again without relying on a second persistent "in progress" state. The persisted queue-state model should stay deliberately small: `waiting` before a run starts, `running` while that queue item owns the active ingest, and `cleanup-blocked` when terminal work finished but the queue record could not yet be removed. There is no separate persisted `abandoned` state for this story; on startup, any leftover `running` record is treated as the abandoned previously-active item and retried.

This story also fixes duplicate-request behavior. If a request arrives for a canonical embed target path that is already waiting in the queue, the server should not insert a second queue item. Instead, it should update that existing waiting item's stored settings in place, keep the same queue request id and queue position, return success, and log that the queued work was intentionally updated rather than duplicated. Those in-place updates should preserve the original queue identity and ordering metadata, including the durable request id, the original `createdAt`, and the original source-surface provenance, while refreshing `updatedAt` and replacing the stored normalized request settings with the latest request. If the matching item is already running, the server should still avoid creating a second queue item, but it should not try to mutate the active run's settings mid-flight. This latest-settings-wins rule for waiting items applies across both start-ingest and re-embed requests once the server has resolved them to the same canonical embed target path.

MongoDB is required for this queue feature. If Mongo is unavailable, queueable ingest and re-embed requests should be rejected before they start. This story does not provide a degraded "run immediately without persistence" mode because that would break the durable queue contract and make restart behavior inconsistent.

Agent and workflow behavior must remain predictable. If a re-embed request is triggered from a flow, command JSON file, or MCP surface that currently waits for terminal completion, that caller should still wait until the queued request actually finishes. Queueing changes how long the caller may need to wait, but it does not change the contract from blocking completion to fire-and-forget. The preferred server behavior is event-driven waiting for the queue item and ingest run to reach a terminal state, not a short fixed timeout loop. A much longer safety timeout can still exist as a final guard for broken listeners or lost state, but it should not be the normal reason a correctly queued request fails. When Mongo-backed queue persistence is unavailable, REST should surface that as `503 Service Unavailable` with `QUEUE_UNAVAILABLE`, while MCP, flows, and commands should preserve the same `QUEUE_UNAVAILABLE` meaning as structured retryable errors rather than flattening it into a generic invalid request.

The frontend must also make queued work visible. When a repository is queued for re-embed, the UI should show that it is queued and show its queue position so the user knows the request was accepted and is waiting its turn. That queue position should count only waiting work, so `1` means “next in line” rather than “behind the currently running item plus everything else.” The repository list is the required source of truth for this state; this story does not need a separate top-of-page queue card. In practice, queued state and queue position should be added to the shared repository-list payload so the existing repo-list mirrors stay aligned, while unrelated flow-facing or generic status payloads do not need their own separate queue-position fields. If a brand-new repository is queued before it has ever been ingested, the repository list should synthesize a temporary row right away so queued start-ingest work is visible there too. This should be a stable status in the ingest surfaces, not just a short-lived toast.

Queue acceptance also changes submission behavior on the ingest page. While another run is active, start-ingest and re-embed actions should still be submittable so they can join the durable queue, rather than remaining disabled just because work is already running. This story does not require new hidden client-side queue fields or a separate local queue cache; the browser should keep submitting the visible current values, and queued visibility should come back from the shared server-side repository-list contract.

The client-side state rules should stay simple. After a queueable submission is accepted, the form may keep its visible values locally for user convenience, but those retained values must not mutate an already accepted queue item unless the user explicitly submits again. Removing the active-run disable gate also does not unlock fields that are disabled for a different reason, such as the existing embedding-model lock.

The queue is FIFO by creation time. On server startup, if the queue collection contains leftover items and no ingest is already active, the server should first retry or resolve any cleanup-blocked queue item that previously failed deletion, and only then attempt to start the oldest remaining waiting item automatically. Because the active item remains in Mongo until terminal completion, this gives the queue at-least-once recovery semantics after restart without needing a second persistent run-state table.

### Acceptance Criteria

- Ingest and re-embed requests use one durable MongoDB queue collection rather than being rejected immediately whenever another ingest run is active.
- Every queueable request is normalized to the canonical embed target path before dedupe and queue insertion decisions are made.
- If MongoDB is unavailable, queueable ingest and re-embed requests are rejected before any new run starts.
- When MongoDB is unavailable, queueable ingest and re-embed requests return a clear retryable `QUEUE_UNAVAILABLE` error rather than a generic server failure.
- The queue uses one collection for both waiting and currently running items; a running item remains in the collection until terminal completion.
- Queue documents persist one coarse queue-state enum with exactly `waiting`, `running`, and `cleanup-blocked`.
- If no ingest is active when a request is accepted, the server inserts the queue item and then starts that oldest queued item immediately.
- If an ingest is already active when a request is accepted, the server inserts the queue item and leaves it waiting in FIFO order.
- Queue order is FIFO by queue creation time.
- On server startup, if no ingest is active and the queue collection contains leftover items, the server attempts to start the oldest remaining item automatically.
- On startup, any leftover `running` queue item is treated as abandoned previously-active work to retry; this story does not require a separate persisted `abandoned` state.
- When an ingest or re-embed run reaches a terminal outcome, the server attempts to delete that queue item before it attempts to start the next queued item.
- If queue-item deletion fails after a terminal run, the server logs an error, shows an error state in the frontend, retries the delete with backoff, and does not start the next queued item until that queue-record removal problem is resolved.
- If a request arrives for a canonical embed target path that is already waiting in the queue, the server does not create a duplicate queue item.
- For a matching waiting queue item, the newer request updates the stored queued settings in place, keeps the same queue request id, and keeps the same queue position.
- For a matching waiting queue item, the record keeps its original `createdAt`, source-surface provenance, and FIFO position while refreshing `updatedAt` and replacing the stored normalized request settings with the latest request.
- If a matching queue item is already running, the server still avoids creating a duplicate queue item, but it does not mutate the active run's settings mid-flight.
- The duplicate-request rule applies across both start-ingest and re-embed requests once they resolve to the same canonical embed target path.
- When a duplicate request is collapsed onto an existing queue item, the server returns success and reuses the existing queue request id instead of returning a duplicate-specific error.
- Queue-aware success responses expose `queued`, `requestId`, and `queuePosition`.
- `requestId` is the durable queue identifier for this story.
- `runId` remains the active ingest-run identifier once work has actually started.
- Queue-aware success responses do not expose a separate `deduped` field in this story.
- For an immediately started request, the success payload includes the queue request id, the active `runId`, and a non-waiting queue state.
- For a waiting request, the success payload includes `queued: true`, the queue request id, the current queue position, and no active `runId` yet.
- `queuePosition` counts only waiting requests; an already running item uses the non-waiting state instead of reporting itself as queue position `1`.
- Flow, command, and MCP re-embed callers that previously blocked until terminal completion still block until the queued request reaches a terminal outcome.
- Queue wait time is treated as part of the blocking contract for those callers rather than as a reason to return early.
- The preferred blocking path waits for real queue-item or ingest completion events rather than relying on a short fixed timeout as the normal control mechanism.
- A much longer safety timeout may still exist as a final guard, but it is not the normal success path for correctly queued work.
- The frontend ingest surfaces show when a repository is queued for re-embed and show its queue position in the repository list.
- Queued state and queue position are added to the shared repository-list payload used by the ingest repo-list surfaces, rather than mirrored into unrelated flow-facing status payloads just for parity.
- The shared repository-list schema version, REST `/ingest/roots` response, classic MCP `ListIngestedRepositories` response schema, and client normalization are updated together so queued rows can expose `requestId`, waiting-only `queuePosition`, and a `runId` that is absent until execution actually starts.
- If a brand-new repository is queued before its first ingest run starts, the repository list still shows a temporary queued row right away.
- Queued repository visibility is stable enough that a user can refresh or revisit the ingest page and still see that the request is waiting.
- While a run is already active, the ingest UI still allows queueable start-ingest and re-embed submissions instead of blocking them behind a global active-run disable state; controls may remain disabled only when they are blocked for a separate destructive or invalid-state reason.
- After a queueable submission is accepted, any later edits to the retained visible form fields affect only a later explicit submit; they do not silently mutate the already accepted queue item.
- Removing the active-run disable gate does not change existing lock-driven field behavior: if the embedding model is locked, the model selector remains read-only and submissions keep using that locked model identity.
- Existing single-flight ingest execution still applies at runtime: only one ingest or re-embed run is active at a time, even though multiple future requests may now be queued durably.
- The queue request id may use the MongoDB document id generated for that queue item.
- On startup, cleanup-blocked queue items are retried or resolved before the server starts any newer waiting item.
- REST queue-backend outages return HTTP `503 Service Unavailable` with `QUEUE_UNAVAILABLE`; MCP, flow, and command surfaces preserve `QUEUE_UNAVAILABLE` as a structured retryable error instead of collapsing it into a generic invalid-request failure.
- This story does not require queued-but-not-started requests to be removable by the user once they are in the queue.
- Documentation and tests are updated to describe the durable queue contract, dedupe behavior, startup recovery, blocking caller behavior, and frontend queued-state visibility.

### Out Of Scope

- User-driven removal or cancellation of queued-but-not-started requests.
- Priority queues, non-FIFO ordering, or queue reordering controls.
- Multiple concurrent ingest workers processing more than one queued request at once.
- Replacing the existing internal file-chunking and embedding-dispatch optimizations planned elsewhere.
- Adding a degraded mode that still starts queueable requests when MongoDB is unavailable.
- Returning a separate `deduped` response field instead of the queue-oriented response contract defined here.
- Broad redesign of the ingest frontend beyond what is needed to show queued state and queue position.
- Adding a dedicated top-of-page queue card for queued re-embed work.
- Adding queue-position fields to unrelated non-repository read surfaces when they are not acting as the story's required source of truth.
- Exactly-once execution guarantees across crashes or restarts.

## Message Contracts And Storage Shapes

- Current route contracts are still single-flight:
  - `POST /ingest/start` returns `202 { runId }` or `429 { status:'error', code:'BUSY' }`.
  - `POST /ingest/reembed/:root` returns `202 { runId }` or `429 { status:'error', code:'BUSY' }`.
  - `GET /ingest/status/:runId` remains the active per-run progress surface.
- This story should replace the write-side success contract for queueable start and re-embed requests with a queue-aware shape that is explicit enough for later tasking:
  - immediately started success: `queued: false`, `requestId`, `runId`, and no waiting queue position;
  - waiting success: `queued: true`, `requestId`, `queuePosition`, and no active `runId` yet;
  - duplicate waiting updates reuse the same `requestId` and queue position instead of returning a duplicate-only error.
- Queue-aware REST failure responses should keep the repo's current error-envelope style rather than inventing a second format. For queue persistence outages, the plan should assume `503` with a JSON body shaped like the existing ingest route failures: `{ status: 'error', code: 'QUEUE_UNAVAILABLE', retryable: true, message: string }`, plus an optional HTTP `Retry-After` header when the server has a meaningful retry delay to suggest.
- The durable queue document should stay intentionally small and explicit. At minimum it must carry:
  - one durable request identity, preferably the Mongo `_id`;
  - canonical target path and operation type;
  - persisted queue state `waiting | running | cleanup-blocked`;
  - created and updated timestamps;
  - the normalized request settings that will later drive the run;
  - enough provenance to explain where the request came from when troubleshooting or proving updates.
- Queue-document ownership should stay explicit in the plan. The shared admission helper is the only write path that creates or updates waiting queue items, the queue pump and startup-recovery paths are the readers that promote waiting items into active execution, the shared repo-list readers may read queue state for visibility but do not mutate it, and the terminal cleanup path is the only code allowed to delete queue documents after completion, error, or cancellation.
- The queue collection should follow the repo's existing Mongoose timestamp pattern (`timestamps: true`) and use indexed selectors for duplicate handling and FIFO reads. Prefer an indexed `findOneAndUpdate(..., { returnDocument: 'after' })` or equivalent single-document operation for duplicate waiting-item reuse, and do not rely on unindexed upsert behavior for atomicity. The canonical target-path selector and queue-order selector must therefore be backed by concrete indexes before later tasking relies on them.
- The shared repository-list contract is currently version `0000038-status-phase-v1`, and its common types are still run-centric: [common/src/lmstudio.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/common/src/lmstudio.ts) exposes only `status` plus `phase`, [server/src/lmstudio/toolService.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/lmstudio/toolService.ts) models rows around one `id`, and [client/src/hooks/useIngestRoots.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/hooks/useIngestRoots.ts) expects a `runId` string on each normalized row. Because a waiting queue item has a durable `requestId` before any `runId` exists, this story must extend the shared repo-list contract in one place instead of overloading `runId`; if the queued row shape needs a new identifier or nullable `runId`, bump the shared schema version and update server plus client consumers together.
- The queue-aware repo-list contract should be stated concretely so later tasking does not guess. A queued row needs an explicit `requestId`, an optional `runId` that stays `null` or absent until execution actually starts, and a waiting-only `queuePosition` that is a positive integer when `queued: true` and absent otherwise. Because the classic MCP `ListIngestedRepositories` schema in [server/src/mcp/server.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/mcp/server.ts) currently hard-codes the older run-centric shape, the schema version bump must update that MCP output schema and the shared client/common types in the same story.
- The repository-list read contract is the required queued-state source of truth for users. Add queued state and waiting-only queue position to the shared repo-list payload that already feeds the ingest page and its mirrors, but do not mirror queue-position fields into unrelated generic status payloads purely for symmetry.
- Queue-backed repository rows, including brand-new roots that have never been ingested, should be synthesized in the shared server list builders ([server/src/lmstudio/toolService.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/lmstudio/toolService.ts) and [server/src/routes/ingestRoots.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/routes/ingestRoots.ts)) before the React hook normalizes them, so REST, MCP, and UI mirrors all consume the same canonical queued view.
- The blocking re-embed surfaces should keep their existing terminal-completion promise, but their internal wait contract must change from today's short poll-timeout bias to a queue-aware wait that treats queue delay as normal behavior.

## Edge Cases And Failure Modes

- The current repo still has hard single-flight `BUSY` checks in the start and re-embed routes plus a `90_000 ms` polling wait in [server/src/ingest/reingestService.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/reingestService.ts). Later tasking must replace those consistently so the product does not end up half-queued and half-single-flight.
- The current client still disables start, re-embed, and remove actions whenever any run is active because [client/src/pages/IngestPage.tsx](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/pages/IngestPage.tsx) passes `disabled={isRunActive}` into both the form and the roots table, and [client/src/components/ingest/RootsTable.tsx](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/components/ingest/RootsTable.tsx) derives row/button disablement from that shared `busy` state. Later tasking must remove only the gating that conflicts with queued acceptance while still preserving genuinely destructive protections.
- Queue dedupe must happen on the canonical target path after normalization, not on raw request text or route params, otherwise start-ingest and re-embed requests for the same repository can still diverge.
- The queue state visible to users must remain waiting-only for `queuePosition`; already-running work should not pretend to be queue position `1`.
- The existing `CODEINFO_INGEST_MAX_QUEUE_SIZE` setting already bounds the embedding dispatcher's internal work queue, so using it for request-level queue admission or delete-retry behavior would couple two different scales incorrectly. Request-queue sizing and cleanup retry behavior should stay code-defined in this story unless fresh repo evidence proves a separate operator-facing control is required.
- Because the shared repo-list types currently assume a run-centric row shape, waiting queue items cannot be represented safely by pretending `requestId === runId`. The plan must keep those identifiers distinct all the way through the shared server and client contract.
- Queue dedupe, startup replay, and queue-position lookups must stay index-backed and bounded as repository counts grow. The plan should avoid any later design that requires scanning the full queue collection by raw path text just to decide whether a request can be inserted or to compute the next waiting position.
- Because the UI may now keep visible form state while queue items already exist, the server must remain authoritative when stale or repeated client input arrives. A repeated submit for the same canonical target should flow through the queue's duplicate-update or running-item-reuse rules rather than relying on the client to prevent mixed state perfectly.
- Active-run cancellation must still flow through the same terminal cleanup ownership as completion and error. If a queued request has already started and its ingest run is cancelled, the owning queue document still needs terminal cleanup before the next waiting item may start; cancellation must not strand a `running` queue record indefinitely.
- Cleanup-blocked records must prevent the queue from advancing on both live runtime and startup recovery paths; otherwise FIFO ordering silently breaks under delete failures.
- Because the repo-list is the required queued-state source of truth, a brand-new queued root needs a synthesized row immediately. Waiting for the first active run to start would make accepted queued work look lost.
- Mongo outages must fail before new work starts. A partial \"queue unavailable but run anyway\" branch would violate the durable-queue contract and create restart behavior the plan explicitly excludes.

## Implementation Ideas

- Server queue-storage seam: add one MongoDB collection for queued ingest work, for example `ingest_queue_requests`, with the Mongo `_id` used as the external `requestId`. Define the document shape, `timestamps: true`, the `waiting | running | cleanup-blocked` state enum, and the smallest useful indexes for canonical-target duplicate lookup and FIFO waiting-order reads.
- Server queue-admission seam: add one shared `enqueueOrReuseIngestRequest(...)` helper that runs after canonical path and embedding-selection resolution, rejects immediately when Mongo is unavailable, reuses the existing waiting item for a duplicate target with an indexed single-document update, reuses the existing running item without mutating it, or inserts a brand-new waiting queue item.
- Server duplicate-update seam: when the admission helper updates an existing waiting item, keep `requestId`, `createdAt`, canonical target path, and source-surface provenance stable; refresh `updatedAt`; replace the normalized request payload instead of deep-merging it; and return the post-update document so later callers do not have to re-read the queue item just to know what now owns the slot.
- Server queue-pump seam: add one shared `pumpIngestQueue()` function that reads the oldest eligible queue item, starts work only when the existing in-memory ingest lock is idle, and creates the real ingest `runId` only at the point where the queued request actually begins execution.
- Server terminal-cleanup seam: keep the active queue document in Mongo while the run is executing, then delete it before advancing the queue. If deletion fails, move the queue item into the `cleanup-blocked` path, expose the blocked error state, and retry cleanup with backoff without starting newer work.
- Server startup-recovery seam: hook queue recovery into the existing boot path in [server/src/index.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/index.ts) after `connectMongo(...)` succeeds and before the server is treated as ready. Resolve any `cleanup-blocked` item first, then retry leftover `running` work, then resume the oldest remaining `waiting` item.
- REST write-contract seam: update the start-ingest and re-embed routes so they stop returning single-flight `BUSY` for queueable requests, emit the queue-aware success shape (`queued`, `requestId`, `runId`, `queuePosition`), and return the explicit `QUEUE_UNAVAILABLE` outage contract when Mongo-backed queue persistence cannot accept the request.
- Blocking-caller seam: update flows, commands, and MCP re-embed callers so they enqueue or reuse a durable queue item first, then wait for that queued request's terminal outcome using queue-aware completion events rather than today's short polling timeout as the normal control path.
- Shared repo-list contract seam: update the shared repository-list data contract, schema version, and classic MCP `ListIngestedRepositories` schema together so queued rows can carry `requestId`, nullable-or-absent `runId` before execution starts, and waiting-only `queuePosition` without overloading the old run-centric shape.
- Server repo-list reader seam: extend [server/src/lmstudio/toolService.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/lmstudio/toolService.ts) and [server/src/routes/ingestRoots.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/routes/ingestRoots.ts) so they overlay waiting queue entries from Mongo alongside active run contexts and Chroma roots, including synthesized rows for brand-new queued roots that do not yet exist in the persisted repo list.
- Client queue-visibility seam: update the ingest-page client normalization and rendering so queued rows consume the shared server-side repo-list contract, show waiting-only queue position, and stop relying on a global active-run disable state for queueable start-ingest and re-embed submissions. Keep queued visibility sourced from the shared repo-list contract rather than a separate local queue cache, keep retained visible form state local-only after a successful submit, and preserve lock-driven read-only field behavior where it already exists.
- Backend proof seam: add or extend server-side proof so the queue-storage, admission, duplicate-update, queue-pump, terminal-cleanup, startup-recovery, and `QUEUE_UNAVAILABLE` paths are all independently provable through unit plus Testcontainers/Cucumber coverage rather than one bundled “queue works” test.
- Frontend and end-to-end proof seam: add or extend client and Playwright proof so queued rows, waiting-only `queuePosition`, brand-new queued roots, and queueable submission behavior while another run is active are all validated separately from the backend persistence and recovery paths.
