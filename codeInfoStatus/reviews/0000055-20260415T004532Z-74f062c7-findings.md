3. `must_fix` | `generic_engineering_issue` | repo scope: `current_repository`
   Queue-managed runs have no durable replay barrier if the post-commit `terminalPublishedAt` write fails, so restart recovery can rerun a logically finished operation and duplicate its side effects. The run finally block writes `terminalPublishedAt` via [persistQueueTerminalBarrier()](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts:586) before queue cleanup and lock release at [server/src/ingest/ingestJob.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts:2525), but that write is not caught locally even though the queue row is the only persisted signal that startup recovery uses to skip replay at [server/src/ingest/ingestJob.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts:2676). If Mongo drops after the ingest or reembed side effects have already committed but before `markQueueRequestTerminalPublished()` succeeds, recovery will still see a `running` row with no `terminalPublishedAt` and resume it as unfinished work instead of treating it as “committed but cleanup not finished”. The current changed tests cover the happy path where `terminalPublishedAt` already exists and recovery skips replay ([server/src/test/unit/ingest-queue-runtime.test.ts](/home/d_a_s/code/codeInfo2/server/src/test/unit/ingest-queue-runtime.test.ts:823)), but I did not find proof for the barrier-write-failure branch; every touched queue-runtime test stub I re-read keeps `markQueueRequestTerminalPublished` as a no-op success.
