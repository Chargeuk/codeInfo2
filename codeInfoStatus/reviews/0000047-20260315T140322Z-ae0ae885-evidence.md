## Story Extraction

### Description

Story 47 unifies Codex chat/runtime config behavior around one predictable runtime path. The intended end state is:

- `codex/chat/config.toml` supplies the default Codex chat model and contributes to the available model list.
- `Codex_model_list` ordering is preserved, with the chat-config model appended only when missing.
- model availability and default resolution reread `codex/chat/config.toml` on each request.
- missing `codex/config.toml` and `codex/chat/config.toml` bootstrap from canonical in-code templates instead of sample files.
- base-only runtime settings still flow into resolved chat and agent runtime config after direct chat-template bootstrap.
- Context7 API-key handling is normalized in memory from `CODEINFO_CONTEXT7_API_KEY`, including placeholder-equivalent keys and the no-key local stdio args form.

### Acceptance Criteria

1. Chat-config model is a first-class available Codex model.
2. Shared Codex capability/model resolution owns merged model-list behavior.
3. Merged model list uses env list union current chat-config model with deterministic dedupe and append-only behavior.
4. Chat-config model is the default Codex chat model unless explicitly overridden.
5. Default precedence is request override -> chat config -> env -> hardcoded fallback.
6. Invalid/unreadable chat config continues to warn and fall back.
7. Invalid/unreadable existing chat config is left in place.
8. Chat config is reread from disk for model/default selection.
9. Web chat model list reflects current chat config.
10. MCP chat/default selection reflects current chat config.
11. `/chat/providers` uses the shared Codex-aware path instead of env-only Codex defaults.
12. Codex-facing server entrypoints, including `codebaseQuestion`, use the same shared Codex-aware behavior.
13. Existing client contract keeps consuming `/chat/providers` and `/chat/models` without a story-specific response-shape change.
14. `CHAT_DEFAULT_MODEL` is fallback-only.
15. Missing `codex/chat/config.toml` bootstraps from one canonical in-code chat template.
16. Missing `codex/config.toml` bootstraps from one canonical in-code base template.
17. Base and chat templates both use `model = "gpt-5.3-codex"`.
18. Canonical in-code templates are the only first-time bootstrap source.
19. `config.toml.example` remains documentation-only.
20. Missing chat bootstrap uses the chat template directly instead of copy-from-base mutation.
21. Missing-file bootstrap does not overwrite existing configs.
22. Shared runtime resolution preserves required base data for chat and agent execution.
23. At minimum, `mcp_servers`, `model_provider`, and `model_providers` remain available unless explicitly overridden.
24. Direct chat-template bootstrap does not remove inherited runtime settings such as MCP servers, tools, personality, and provider-routing config.
25. Runtime config applies `CODEINFO_CONTEXT7_API_KEY` in memory when a local stdio Context7 definition exists and no usable key is present.
26. Placeholder and legacy seeded Context7 key values count as unusable.
27. Placeholder-equivalent values are overlaid from env when the env key is usable.
28. Placeholder-equivalent values fall back to the no-key args form when the env key is missing or empty.
29. No-key fallback removes or replaces only the `--api-key` pair and preserves unrelated args and unrelated MCP definitions.
30. Explicit non-placeholder Context7 keys win.
31. Context7 normalization applies only to local stdio `command` plus `args` definitions.
32. Context7 normalization does not rewrite config files on disk.
33. Context7 normalization applies consistently to chat and agent/runtime config loading.
34. Unrelated MCP args and unrelated config values are preserved.
35. Canonical templates no longer depend on checked-in Context7 key material.
36. Public REST and MCP payload shapes remain unchanged.

- remote/header-based Context7 normalization
- generic credential validation or migration frameworks
