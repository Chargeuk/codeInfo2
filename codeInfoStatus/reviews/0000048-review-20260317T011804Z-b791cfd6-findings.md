### 1. Chat requests still accept non-ingested working folders, which breaks the story’s shared working-folder contract

- Classification: `must_fix`
- Issue type: `plan_contract_issue`
- Files:
  - `server/src/routes/chatValidators.ts:239`
  - `server/src/routes/chat.ts:384`
  - `server/src/routes/chat.ts:391`
  - `server/src/routes/chat.ts:543`
- Why this matters:
  - The plan explicitly says chat must use the same canonical absolute repository-root contract as agents, flows, and commands, including the “invalid because no longer ingested” behavior.
  - Agents, flows, and the new conversation-edit route all pass `knownRepositoryPaths` into `validateRequestedWorkingFolder(...)`, but chat validation does not. `POST /chat` therefore still accepts any absolute existing directory, even if it is not one of the ingested repositories that the rest of the story now treats as valid working folders.
  - Once accepted, `chat.ts` trusts `requestedWorkingFolder` and persists/logs it without re-checking ingested-repo membership.
- Malformed / contradictory input to test explicitly:
  - `POST /chat` with `working_folder` set to an existing absolute path that is outside the ingested repository set, while the same path would be rejected by `POST /conversations/:id/working-folder`.
- Current proof:
  - Missing direct proof for the reject case; the coverage here is happy-path only.
- Before/after contract comparison:
  - `POST /chat` is an additive surface for `working_folder`, but its new contract is weaker than the story contract and weaker than the other Story 48 entry points. This is not a permitted compatibility deviation in the plan.

### 2. Saved working folders are cleared as “stale” for any `fs.stat` failure, including unreadable or transient filesystem errors

- Classification: `should_fix`
- Issue type: `generic_engineering_issue`
- Files:
  - `server/src/workingFolders/state.ts:35`
  - `server/src/workingFolders/state.ts:118`
  - `server/src/workingFolders/state.ts:183`
- Why this matters:
  - `isDirectory(...)` converts every `fs.stat(...)` failure into `null`, which means permission errors, transient I/O failures, and other non-“not found” conditions are all collapsed into `WORKING_FOLDER_NOT_FOUND`.
  - `restoreSavedWorkingFolder(...)` then catches that generic failure, clears the persisted folder, and logs a stale-path warning. That matches the plan only for genuinely missing or no-longer-ingested paths, not for unreadable or temporarily unavailable ones.
  - The result is a destructive partial-failure path: an operational filesystem problem becomes silent data loss plus an inaccurate “saved_value_invalid” warning instead of an actionable diagnostic.
- Malformed / contradictory input to test explicitly:
  - A previously valid saved folder whose directory exists but is temporarily unreadable or whose mapped workdir path throws a non-ENOENT filesystem error.
- Current proof:
  - Missing direct proof for non-ENOENT stat failures; current tests cover relative paths, successful mappings, and missing directories only.
- Before/after contract comparison:
  - The new stale-folder clearing contract is additive and intended, but the implementation currently broadens “invalid” into “any stat failure,” which is not clearly allowed by the active plan’s invalid-path contract.

### 3. Repository de-duplication lowercases every resolved path, so distinct repositories can collapse on case-sensitive filesystems

- Classification: `should_fix`
- Issue type: `generic_engineering_issue`
- Files:
  - `server/src/flows/repositoryCandidateOrder.ts:37`
  - `server/src/flows/repositoryCandidateOrder.ts:48`
  - `server/src/test/unit/repositoryCandidateOrder.test.ts:63`
- Why this matters:
  - `normalizeCandidateKey(...)` lowercases `path.resolve(sourceId)` before using it as the de-duplication key.
  - That matches the current happy-path tests, but it also means `/Repos/Game` and `/Repos/game` are treated as the same repository even on case-sensitive filesystems where they can be distinct roots.
  - The plan required duplicate absolute repository identities to be removed; it did not authorize case-folding distinct repository paths into one candidate. On Linux/container paths this can silently delete a valid candidate and alter fallback order.
- Malformed / contradictory input to test explicitly:
  - Working repo `/tmp/Repo-One` and owner repo `/tmp/repo-one` on a case-sensitive filesystem where both directories are real and distinct.
- Current proof:
  - Direct proof exists only for the case-insensitive de-duplication behavior; there is missing proof for the case-sensitive “distinct repos must both survive” path.
- Before/after contract comparison:
  - The new helper centralizes precedence correctly, but its de-duplication behavior is broader than the plan’s “same absolute path” requirement.

### 4. The new shared client runtime-config helper silently treats malformed canonical config as success by defaulting it away

- Classification: `should_fix`
- Issue type: `generic_engineering_issue`
- Files:
  - `client/src/config/runtimeConfig.ts:49`
  - `client/src/config/runtimeConfig.ts:60`
  - `client/src/config/runtimeConfig.ts:104`
  - `client/src/test/baseUrl.env.test.ts:1`
  - `client/src/test/logging/transport.test.ts:1`
- Why this matters:
  - `normalizeBoolean(...)` and `normalizeNumber(...)` return `undefined` for malformed canonical values, and `getClientRuntimeConfig()` immediately replaces them with env/default values.
  - That means invalid `VITE_CODEINFO_LOG_FORWARD_ENABLED`, `VITE_CODEINFO_LOG_MAX_BYTES`, runtime-injected booleans/numbers, and empty-string URL fields are silently normalized into a valid-looking config with no warning marker or rejection path.
  - The new helper was introduced specifically to make runtime/build config behavior clearer and safer. Silently swallowing malformed canonical inputs makes misconfiguration harder to detect and weakens the reviewability of the new `DEV_0000048_T8_VITE_CODEINFO_RUNTIME_CONFIG` marker.
- Malformed / contradictory input to test explicitly:
  - Runtime config sets `{ logForwardEnabled: 'maybe', logMaxBytes: 'zero', apiBaseUrl: '   ' }` while env also provides canonical values; verify whether the runtime marker exposes the invalid override or whether it is silently replaced by fallback/default.
- Current proof:
  - Missing direct proof for malformed canonical values; current tests cover happy-path canonical names and legacy-name rejection, not invalid canonical-value handling.
- Before/after contract comparison:
  - The env-name rename is intentionally breaking and allowed by the plan. The risky change here is separate: the new shared runtime-config helper now merges runtime/env/default sources before any invalid-canonical-value warning or rejection can happen.
