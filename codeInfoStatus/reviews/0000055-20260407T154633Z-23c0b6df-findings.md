# Review Findings

## Findings

1. `must_fix` `generic_engineering_issue` `scope: current_repository`  
   The new delta no-op re-embed fast path can bypass the promotion-time execution validation that queued work is supposed to re-check. In [server/src/ingest/ingestJob.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts#L1460), `processRun()` returns through `completeReembedFastPathWithFence()` before it reaches `validateExecutableIngestInput()` at [server/src/ingest/ingestJob.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts#L1480). That means a queued re-embed which becomes zero-work can finish as `completed` even if the locked embedding model changed or the requested OpenAI model is no longer executable by the time the queued item actually starts. The intended invariant is explicit in the normal promotion test at [server/src/test/unit/ingest-queue-runtime.test.ts](/home/d_a_s/code/codeInfo2/server/src/test/unit/ingest-queue-runtime.test.ts#L202), but the current no-op tests only prove provider-free/bootstrap-free success at [server/src/test/unit/ingest-reembed.test.ts](/home/d_a_s/code/codeInfo2/server/src/test/unit/ingest-reembed.test.ts#L475) and [server/src/test/unit/ingest-reembed.test.ts](/home/d_a_s/code/codeInfo2/server/src/test/unit/ingest-reembed.test.ts#L551), not the contradictory no-op-plus-lock-mismatch case.
