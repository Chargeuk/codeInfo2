### 1. Invalid `/chat/models` provider values are still coerced into a successful LM Studio response

- Severity: `must_fix`
- Issue type: `plan_contract_issue`
- Repo scope: `current_repository` (`/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2`)

The story expands the chat-model surface into a three-provider contract and keeps provider selection deterministic across shared routes, but [`server/src/routes/chatModels.ts`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/routes/chatModels.ts#L177) still behaves as "not `codex` and not `copilot` means LM Studio". This breaks the planned contract in [`planning/0000051-github-copilot-sdk-chat-provider.md`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/planning/0000051-github-copilot-sdk-chat-provider.md#L95) and [`planning/0000051-github-copilot-sdk-chat-provider.md`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/planning/0000051-github-copilot-sdk-chat-provider.md#L116): the route is meant to expose only the shared provider enum and to extend the existing contract to include `copilot`, not silently reinterpret invalid provider values as LM Studio success. The current test coverage in [`server/src/test/unit/chatModels.copilot.test.ts`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/test/unit/chatModels.copilot.test.ts#L110) proves the happy-path Copilot mapping, but there is no direct proof that malformed `provider` query values are rejected.

Proof status: missing direct proof for malformed-provider rejection.

### 2. `useChatModel` hides provider and model fetch failures behind a synthetic LM Studio success state

- Severity: `must_fix`
- Issue type: `plan_contract_issue`
- Repo scope: `current_repository` (`/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2`)

The plan requires unavailable providers to stay visible with stable reasons and for readiness fields such as `available`, `toolsAvailable`, and warnings to stay distinct. Instead, [`client/src/hooks/useChatModel.ts`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/hooks/useChatModel.ts#L69) manufactures a degraded provider list where LM Studio is always marked available, and both the provider-fetch catch path and the model-fetch catch path then promote that degraded state into a full UI success state: [`client/src/hooks/useChatModel.ts`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/hooks/useChatModel.ts#L262), [`client/src/hooks/useChatModel.ts`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/hooks/useChatModel.ts#L378). On any `/chat/providers` or `/chat/models` failure, the hook sets `available=true`, `toolsAvailable=true`, `status='success'`, and inserts a fake fallback model even though the live backend request just failed.

That behavior contradicts the planned provider contract in [`planning/0000051-github-copilot-sdk-chat-provider.md`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/planning/0000051-github-copilot-sdk-chat-provider.md#L97) and [`planning/0000051-github-copilot-sdk-chat-provider.md`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/planning/0000051-github-copilot-sdk-chat-provider.md#L103): the page is supposed to keep unavailable providers visible with stable reasons, not silently reframe a transport or server failure as a healthy LM Studio bootstrap. Proof status: direct proof that the degraded-success path exists, but missing proof for correct failure visibility.

### 3. Copilot device-auth can still escape as a raw server error before it reaches the shared unavailable-before-start contract

- Severity: `must_fix`
- Issue type: `plan_contract_issue`
- Repo scope: `current_repository` (`/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2`)

`createCopilotDeviceAuthRouter` was one of the evidence step's highest-risk helpers, and there is still an uncovered bootstrap hole in it. The route correctly translates `ensureCopilotAuthFileStore()` failures into `state: 'unavailable_before_start'`, but it calls `ensureCopilotPlaintextTokenStorage()` earlier and outside that protection in [`server/src/routes/copilotDeviceAuth.ts`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/routes/copilotDeviceAuth.ts#L306). The storage helper itself can throw on malformed JSON, `mkdir`, `readFile`, or `writeFile` failures in [`server/src/config/copilotConfig.ts`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/config/copilotConfig.ts#L148). In those cases the request bypasses the route's shared "unavailable before start" response and will surface as a raw Express 500 instead of the deterministic auth-state payload the story planned.

That breaks the auth contract described in [`planning/0000051-github-copilot-sdk-chat-provider.md`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/planning/0000051-github-copilot-sdk-chat-provider.md#L74) and [`planning/0000051-github-copilot-sdk-chat-provider.md`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/planning/0000051-github-copilot-sdk-chat-provider.md#L245): the route is supposed to surface device-flow verification details or a deterministic failure reason, not leak bootstrap exceptions from the underlying Copilot home preparation. The current unit coverage in [`server/src/test/unit/copilotDeviceAuth.test.ts`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/test/unit/copilotDeviceAuth.test.ts#L185) proves the later file-store failure mapping, but it does not exercise plaintext-storage bootstrap failure, so this lifecycle-sensitive path still has only happy-path proof.

Proof status: missing direct proof for plaintext-storage bootstrap failure handling.

### 4. Copilot tool completion events blank out the tool name that the client already recorded for the same call

- Severity: `should_fix`
- Issue type: `generic_engineering_issue`
- Repo scope: `current_repository` (`/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2`)

The story translates Copilot SDK events into the existing shared tool-call contract, but [`server/src/chat/interfaces/ChatInterfaceCopilot.ts`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/chat/interfaces/ChatInterfaceCopilot.ts#L290) emits `tool-result` with `name: ''` on `tool.execution_complete`. The client merge logic in [`client/src/hooks/useChatStream.ts`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/hooks/useChatStream.ts#L766) uses `event.name ?? existing?.name`, so the empty string is treated as authoritative and overwrites the previously stored request name instead of preserving it. The result is that completed Copilot tool calls can lose their name in the transcript or tool details even though the matching `tool-request` event already supplied it.

That violates the shared event-translation intent in [`planning/0000051-github-copilot-sdk-chat-provider.md`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/planning/0000051-github-copilot-sdk-chat-provider.md#L242) and weakens operational debugging because request and result for the same call no longer share the same identifying field vocabulary. The unit coverage in [`server/src/test/unit/chat-interface-copilot.test.ts`](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/test/unit/chat-interface-copilot.test.ts#L56) checks event ordering, but it does not assert that tool names survive the request-to-result transition.

Proof status: missing direct proof for tool-name preservation across request and result events.
