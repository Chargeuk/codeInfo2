7. `must_fix` `generic_engineering_issue` `current_repository`
   Startup recovery can replay a logically completed ingest and duplicate its side effects after a crash. The worker only removes the queue row in `processRun()`'s `finally` block after terminal status publication and other ingest side effects have already happened ([server/src/ingest/ingestJob.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts:2476)), but [recoverIngestQueueOnStartup()](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts:2623) blindly treats any persisted `running` row as unfinished work and calls `startManagedRun()` again with the old request payload. There is no compare-and-swap, completion marker, or terminal-state check to distinguish “started but not committed” from “committed and crashed before queue cleanup”, so a restart after vectors, root metadata, logs, or provider calls have already been committed can run the ingest a second time. The changed tests even codify this replay behavior as success for any leftover running row ([server/src/test/unit/ingest-queue-runtime.test.ts](/home/d_a_s/code/codeInfo2/server/src/test/unit/ingest-queue-runtime.test.ts:797), [server/src/test/integration/ingest-reembed.test.ts](/home/d_a_s/code/codeInfo2/server/src/test/integration/ingest-reembed.test.ts:52)) rather than proving an idempotency barrier for the committed-before-crash case.
