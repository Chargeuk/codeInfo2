# Review Findings

## Findings

### 1. `must_fix` `generic_engineering_issue` `repo_scope=current_repository`

Queued re-embed admission now rewrites the persisted queue target from the repository's canonical stored root to the mounted workdir path, but runtime re-embed bookkeeping still keys delta lookup and cleanup by the discovered `root`. In `buildQueuedReingestRequest()`, both `canonicalTargetPath` and `requestPayload.path` are set to `resolveMountedIngestPath(...)` [reingestService.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/reingestService.ts#L401), while `processRun()` later uses that remapped path as `root` for `listIngestFilesByRoot(root)` and `deleteVectors({ where: { root } })` / `deleteRoots({ where: { root } })` [ingestJob.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts#L831), [ingestJob.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts#L898), [ingestJob.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts#L1472). For legacy-listed roots such as `/data/...` that are remapped to `/home/...`, this can miss the existing Mongo delta index and stale Chroma metadata under the old root, effectively turning a re-embed into a fresh ingest under a new root key while leaving old records behind. The current unit proof only asserts that the remap happens [reingestService.test.ts](/home/d_a_s/code/codeInfo2/server/src/test/unit/reingestService.test.ts#L521); it does not prove old-root cleanup still happens.

External comment decision: endorsed finding. The external comment strengthened the highest-risk helper from the evidence pass and exposed a concrete correctness bug rather than a style concern.

### 2. `should_fix` `plan_contract_issue` `repo_scope=current_repository`

The shared repo-list builder can still emit duplicate rows for the same repository path after repeated ingests, and only one of those rows receives queue/active overlays. `listIngestedRepositories()` first builds `repos[]` from every persisted metadata row [toolService.ts](/home/d_a_s/code/codeInfo2/server/src/lmstudio/toolService.ts#L875), then builds `repoBySourceId` as a last-write-wins map [toolService.ts](/home/d_a_s/code/codeInfo2/server/src/lmstudio/toolService.ts#L951). Queue overlays and active overlays mutate the mapped entry only [toolService.ts](/home/d_a_s/code/codeInfo2/server/src/lmstudio/toolService.ts#L969), [toolService.ts](/home/d_a_s/code/codeInfo2/server/src/lmstudio/toolService.ts#L987), but the final response still serializes the full `repos[]` array without deduping [toolService.ts](/home/d_a_s/code/codeInfo2/server/src/lmstudio/toolService.ts#L1067), and `GET /ingest/roots` now forwards that list directly [ingestRoots.ts](/home/d_a_s/code/codeInfo2/server/src/routes/ingestRoots.ts#L163). That breaks the story's "repository list is the source of truth" contract because users can see stale/conflicting rows for one path. Existing route-side tests still cover only the old helper `dedupeRootsByPath()` in isolation [ingest-roots-dedupe.test.ts](/home/d_a_s/code/codeInfo2/server/src/test/unit/ingest-roots-dedupe.test.ts#L58), not the live `listIngestedRepositories()` path after this refactor.

External comment decision: endorsed finding. The external comment correctly identified a user-visible repo-list regression and maps directly to the plan's shared repository-list acceptance criteria.

### 3. `should_fix` `plan_contract_issue` `repo_scope=current_repository`

`RootsTable.doReembed()` accepts any `2xx` body as success and falls back to "Re-embed started" even when the response does not prove either queue admission or active execution. The new success path only checks `res.ok`, optionally calls `onRunStarted` if `runId` exists, and otherwise shows success unconditionally [RootsTable.tsx](/home/d_a_s/code/codeInfo2/client/src/components/ingest/RootsTable.tsx#L160). By contrast, `IngestForm` now rejects responses missing `requestId` before surfacing acceptance [IngestForm.tsx](/home/d_a_s/code/codeInfo2/client/src/components/ingest/IngestForm.tsx#L241). Because Story 55's queue-aware write contract requires `requestId` plus either queued metadata or a live `runId`, the table can now show a false-positive success state for malformed `2xx` responses. Current table tests cover only the happy-path `runId` response [ingestRoots.test.tsx](/home/d_a_s/code/codeInfo2/client/src/test/ingestRoots.test.tsx#L121), so this path has no direct proof.

External comment decision: endorsed finding. This is a valid low-risk consistency defect in a file already changed by the story, and it should be treated as `should_fix` rather than deferred cleanup.
