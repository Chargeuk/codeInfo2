Ensure any task that updates the front end has detail within the Manual Playwright-MCP testing guidance to take a screenshot that is checked by the manual testing agent to ensure the GUI meets the expectations set within the task. Note that this is only needed IF it makes sense to take a screenshot because the change will be visual, otherwise this can be ignored and a screenshot check should not be needed.

If the story explicitly names design-target assets intended as implementation references, treat that as `Design Contract Present`. In that case, screenshot guidance must also name the exact design asset that each retained screenshot must be compared against. Screenshot capture alone is not sufficient; comparison notes are part of the proof.

This is especially important for the last overall story check task IF we have visual changes that can be checked, because that final task must ask the manual tester to try to capture screenshots for the entire implemented frontend surface area of the story, not only the newest local screen. Ensure that the Manual Testing Guidance says Playwright MCP screenshots are first captured with a relative staging filename in the Playwright output directory, then transferred into the target repository's task-scoped scratch destination `codeInfoTmp/manual-testing/<story-number>/<task-number>/`. `$CODEINFO_ROOT/playwright-output-local` is only a harness-side staging location when the active runtime exposes that bind, not the final destination for cross-repository proof. Task-level artifacts must not be committed from `codeInfoTmp/`; later story closeout may promote a curated subset into `codeInfoStatus/manual-proof/<story-number>/` as durable final story proof.

When screenshots are relevant, the manual tester should always try to capture them. If honest screenshot capture is blocked by tooling or environment limits, record that limitation honestly, but do not treat missing screenshots by themselves as a visual-review finding.
