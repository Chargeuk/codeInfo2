# Review Findings

## Must Fix

1. `must_fix` `plan_contract_issue` `repo_scope: current_repository`
   [`server/src/ingest/reingestService.ts:404`](server/src/ingest/reingestService.ts:404) builds queued re-embed payloads with `requestPayload.name` sourced from `repo.id` before falling back to the canonical path basename, but [`server/src/lmstudio/toolService.ts:1159`](server/src/lmstudio/toolService.ts:1159) overwrites `existing.id` with `active.runId` for active overlays and [`server/src/lmstudio/toolService.ts:1196`](server/src/lmstudio/toolService.ts:1196) synthesizes `id` from `active.runId` for active-only rows. That means queueing a re-embed while the repo list is overlaying an active run can persist a runtime `runId` as the repository `name`, corrupting the stable display and metadata field the story expects to preserve across queueing and retries.

2. `must_fix` `plan_contract_issue` `repo_scope: current_repository`
   [`client/src/components/ingest/RootsTable.tsx:68`](client/src/components/ingest/RootsTable.tsx:68) correctly treats `status === 'ingesting'` as a non-selectable destructive state, but the row-level Remove button is gated only by `hasActiveRun` and `blocksUserRemove(root)` at [`client/src/components/ingest/RootsTable.tsx:404`](client/src/components/ingest/RootsTable.tsx:404), and `blocksUserRemove()` ignores `status === 'ingesting'` when `queueState` is absent. The test added at [`client/src/test/ingestRoots.test.tsx:402`](client/src/test/ingestRoots.test.tsx:402) proves this “active ingest head with no queueState” shape is reachable and kept out of bulk removal, so the single-row Remove button now bypasses the same destructive-state gate after refresh or stale active-run status. That contradicts the story’s control-gating rule that destructive actions may remain disabled for separate invalid-state reasons while queueable actions stay available.

## Should Fix

3. `should_fix` `generic_engineering_issue` `repo_scope: current_repository`
   [`client/src/components/ingest/RootsTable.tsx:160`](client/src/components/ingest/RootsTable.tsx:160) and [`client/src/components/ingest/RootsTable.tsx:205`](client/src/components/ingest/RootsTable.tsx:205) catch their own failures and only update row-local status, but [`client/src/components/ingest/RootsTable.tsx:229`](client/src/components/ingest/RootsTable.tsx:229) still unconditionally reports `Finished selected actions` and clears the whole selection after the loop. In a mixed-success bulk operation, the batch-level state therefore claims full success and discards the selected rows even when one or more item actions failed. Current tests cover which rows are targeted for bulk remove, but they do not prove mixed-success reporting or retry preservation for bulk failures.
