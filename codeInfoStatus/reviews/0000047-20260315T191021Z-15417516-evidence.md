## Plan Extract

### Description

Story 47 unifies Codex chat/runtime behavior around one shared configuration path. The intended outcome is that `codex/chat/config.toml` drives the default Codex chat model and is always unioned into the available Codex model list without disturbing environment-list order, missing `codex/config.toml` and `codex/chat/config.toml` bootstrap from canonical in-code templates instead of checked-in samples, shared base runtime settings still flow into chat and agent runtime config after chat bootstrap becomes minimal, and Context7 API key handling is normalized in memory from `CODEINFO_CONTEXT7_API_KEY` rather than by rewriting files on disk.

The story also explicitly constrains edge cases: unreadable or invalid existing config files must warn and fall back instead of being silently replaced, bootstrap applies only to missing files, and Context7 normalization is limited to the existing local stdio `command` + `args` shape already used by this repository.

### Acceptance Criteria

There are 36 acceptance criteria in the active plan. They cover four major areas:

1. Shared Codex model/default resolution across REST and MCP entrypoints.
2. Canonical bootstrap for missing Codex base/chat config files.
3. Shared base-runtime inheritance after chat bootstrap becomes minimal.
4. In-memory Context7 API-key normalization without rewriting config files.

### Out Of Scope

- Changing agent prompts, MCP tool descriptions, or repository guidance text.
- Adding new workflow step types or changing command/flow execution behavior.
- Rewriting user-edited config files on each request to inject environment values.
- Broad provider-selection behavior changes beyond the default-model/model-list fixes in scope.
- Changing invalid-config fallback behavior from warnings-and-fallbacks to a new hard-error contract.
- Reordering unrelated MCP args or unrelated model entries beyond the explicit merge rules.
- Remote/header-based Context7 normalization beyond the local stdio `args` shape.
- A general-purpose config migration or credential-validation framework.
