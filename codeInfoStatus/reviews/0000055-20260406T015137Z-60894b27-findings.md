## Rejected Risk Notes

- `enqueueOrReuseIngestRequest()` in `current_repository`
  Candidate risk: two concurrent first-submit requests for the same canonical target could collapse incorrectly and overwrite queue identity or original provenance.
  Disposition: `rejected_risk`.
  Evidence: the implementation only overwrites `operation` and `requestPayload` when reusing a waiting row, not `sourceSurface`, `createdAt`, or queue identity ([server/src/ingest/requestQueue.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/requestQueue.ts#L109), [server/src/ingest/requestQueue.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/requestQueue.ts#L163)). The concurrent race test proves a single request id is reused, the second request updates payload, and the original `sourceSurface` remains on the surviving row ([server/src/test/unit/ingest-request-queue.test.ts](/home/d_a_s/code/codeInfo2/server/src/test/unit/ingest-request-queue.test.ts#L284)).
  Remaining weak proof: the direct proof is good for the tested two-request race, but it does not explore larger multi-writer interleavings.

- `waitForQueueRequestTerminalStatus()` in `current_repository`
  Candidate risk: setup-read failures or timeout-fallback failures could leak listeners or escape as raw errors instead of the queue wait contract.
  Disposition: `rejected_risk`.
  Evidence: the helper centralizes cleanup in `settle()`, unregisters the listener and timer on every settled path, catches timeout-fallback read failures, and intentionally keeps the listener/timer alive when the initial read fails so the bounded timeout path can still classify the result ([server/src/ingest/ingestJob.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts#L2511), [server/src/ingest/ingestJob.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts#L2545), [server/src/ingest/ingestJob.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts#L2585)). Service-level tests verify zero dangling listeners and stable `WAIT_TIMEOUT` classification for both timeout-fallback and setup-read failures ([server/src/test/unit/reingestService.test.ts](/home/d_a_s/code/codeInfo2/server/src/test/unit/reingestService.test.ts#L592), [server/src/test/unit/reingestService.test.ts](/home/d_a_s/code/codeInfo2/server/src/test/unit/reingestService.test.ts#L614), [server/src/test/unit/reingestService.test.ts](/home/d_a_s/code/codeInfo2/server/src/test/unit/reingestService.test.ts#L657)).
  Remaining weak proof: the exact same-tick race between a terminal event and the timeout recovery path is still only indirectly covered.

- `recoverIngestQueueOnStartup()` in `current_repository`
  Candidate risk: startup recovery could skip past an older `cleanup-blocked` row with missing `runId` and incorrectly advance to newer work.
  Disposition: `rejected_risk`.
  Evidence: the function returns immediately after logging when the oldest cleanup-blocked row has no `runId`, before it looks at running or waiting work ([server/src/ingest/ingestJob.ts](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts#L2354)). The queue-runtime test proves no running lookup or waiting promotion occurs in that condition ([server/src/test/unit/ingest-queue-runtime.test.ts](/home/d_a_s/code/codeInfo2/server/src/test/unit/ingest-queue-runtime.test.ts#L382)).
  Remaining weak proof: none beyond the normal single-repository happy-path limits already captured in the evidence artifact.
