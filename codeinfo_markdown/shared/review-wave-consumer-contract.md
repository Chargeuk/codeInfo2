# Review-Wave Consumer Contract

When the current story has `codeInfoTmp/reviews/<story-id>-current-review-set.json`, that manifest and the matching `current-review-wave-validation.json` are the authoritative review handoff.

- Require exact `story_id`, `review_wave_id`, `parent_execution_id`, and `targets_sha256` agreement. When review disposition state declares `review_phase`, require the manifest and matching wave validation to declare the same phase. Never rediscover a wave or pointer by timestamp.
- Require every consumed target-local `job_results` entry to have `status` `completed` or `partial`, a non-null `validation_file`, a non-null server-owned `validation`, `validation.usable: true`, and exact `validation.pointer_key`, `validation.pointer_file`, target `target_id`, target `repo_alias`, `review_wave_id`, `story_id`, `plan_path`, `review_session_id`, `review_pass_id`, `parent_execution_id`, `head_commit`, and `comparison_base_commit` agreement with its prepared base, review-set target, and wave. For Open Code results, use only `validation.usable_bundle_ids`; never infer bundle usability from the reviewer pointer. Consume only those jobs plus the manifest's `aggregated_findings`. Keep each finding's `target_ids`, sources, and severity conflicts visible.
- In wave mode, do not require or read the legacy plan-host `current-review-validation.json`; the per-job validation embedded in the matching finalized review set and wave-validation artifact is authoritative. Require `current-review-validation.json` only when no review-set manifest exists and the calling prompt follows the legacy single-repository path.
- Treat missing, failed, stale, and invalid cells as coverage, not as proof of no findings. A fast wave must set `cross_repository_required: true` and cannot close cleanly unless its cross-repository result is usable and `closeout_allowed` is true. A slow wave must set `cross_repository_required: false`, must contain only the expected target-local main-review jobs, and may close without a cross-repository result only when every expected slow job is usable and `closeout_allowed` is true.
- Use the target owner recorded in the manifest for implementation task ownership. Cross-repository findings may require sequenced tasks, but each implementation task still names one repository owner and retains cross-target validation.
- A retry after fixes starts with a newly prepared target snapshot, review wave ID, target HEADs, bases, and pointers. Do not reuse a previous wave's identity or stable result.
- The cross-repository review runs concurrently with every fast target-local wave and is intentionally absent from the final slow target-only wave. Do not add another unconditional cross-repository review after either join; request a follow-up only for explicitly missing or unusable fast-phase coverage.

If no review-set manifest exists, retain the legacy single-repository behavior described by the calling prompt.
