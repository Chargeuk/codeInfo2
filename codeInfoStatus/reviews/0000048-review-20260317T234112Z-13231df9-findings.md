- What is wrong:
  - Task 43 unified the `DEV_0000048_T1_REPOSITORY_CANDIDATE_ORDER` envelope, but the shared builder still defaults `referenceType` to `null`.
  - The flow-command and direct-command emitters call that builder without setting a command reference type, while the markdown resolver sets `referenceType: 'markdownFile'`.
  - The current tests explicitly lock in `referenceType: null` for command-resolution emitters.
- Why this matters:
  - Story 48’s acceptance contract says structured lookup logs are the canonical debugging surface and must include the reference type.
  - With the current value vocabulary, downstream consumers still have to infer command-vs-markdown lookups from `caller` or from a second marker instead of reading the normalized canonical marker directly.
  - This is not just a stylistic cleanup; it leaves the shared log marker semantically weaker on the command surfaces than the plan requires.
- Before/after contract comparison:
  - Before Task 43: the shared marker had inconsistent field sets across emitters.
  - After Task 43: the field set is consistent, but the value vocabulary still diverges semantically because markdown logs identify the lookup type while command logs intentionally do not.
  - Backward compatibility: changing `referenceType` from `null` to an explicit command vocabulary such as `'commandFile'` is a low-risk log-schema tightening in files already changed by the story. It does not broaden public API payloads.
- Recommended fix direction:
  - Keep the shared helper, but pass an explicit command reference type for flow-command and direct-command emitters.
  - Update the existing log-schema assertions so the canonical marker proves that all lookup surfaces publish a meaningful `referenceType`, not just a nullable placeholder.
