# Findings: Story 0000055

## Findings

1. `should_fix` `generic_engineering_issue` `current_repository`  
   `waitForQueueRequestTerminalStatus()` still has an unhandled setup-read failure path. It performs `await resolveQueueRequestRunState(requestId)` before registering its listener or entering the timeout recovery logic, so a transient queue lookup failure at startup rejects the whole blocking wait immediately instead of degrading to the documented timeout/terminal contract. That rejection bubbles out of `runReingestRepository()` as a raw error because the caller only remaps `NOT_FOUND`, validation, and `QUEUE_UNAVAILABLE` codes. The current proof only covers the timeout-fallback rejection path, not this initial read failure.  
   Evidence: `server/src/ingest/ingestJob.ts:2466-2475`, `server/src/ingest/reingestService.ts:496-590`, `server/src/test/unit/reingestService.test.ts:611-655`.

2. `should_fix` `plan_contract_issue` `current_repository`  
   Startup recovery can skip a persisted `cleanup-blocked` row if its `runId` is missing, then continue on to `running`/`waiting` recovery and potentially start newer work. That conflicts with the Story 55 contract that cleanup-blocked items must be resolved before any newer waiting item starts. The inconsistency is concrete in the current codebase: the persisted model allows `runId: null`, `markQueueRequestCleanupBlocked()` accepts `runId: string | null`, and `pumpIngestQueue()` stalls on any cleanup-blocked row, but `recoverIngestQueueOnStartup()` only blocks when the row also has a non-empty `runId`. Existing recovery proof only covers the non-null `runId` case.  
   Evidence: `planning/0000055-users-can-queue-ingest-and-re-embed-requests.md:76`, `server/src/mongo/ingestQueueRequest.ts:20-35`, `server/src/ingest/requestQueue.ts:274-289`, `server/src/ingest/ingestJob.ts:2313-2327`, `server/src/test/unit/ingest-queue-runtime.test.ts:293-325`.

3. `should_fix` `plan_contract_issue` `current_repository`  
   The branch adds a new token-counting CLI and sample payload outside the Story 55 plan surface, but there is no plan task, acceptance criterion, or proof home tying that utility to the ingest-queue story. Because these are non-support files, they should not ride along implicitly: they either need explicit story justification and proof, or they should move to a separate scoped change.  
   Evidence: `package.json:15`, `server/scripts/count-tool-tokens.mjs:1-40`, `scripts/token-tools-sample.json:1-30`, and the canonical Story 55 plan at `planning/0000055-users-can-queue-ingest-and-re-embed-requests.md` remains scoped to ingest queueing, related proof repairs, wrapper/runtime baseline work, and config drift rather than tool-token analysis.

## Rejected Risk Notes

- `enqueueOrReuseIngestRequest()`  
  Candidate challenge: concurrent first-submit requests for the same canonical target from different queueing surfaces still creating two live rows.  
  Decision: rejected risk.  
  Evidence: `server/src/mongo/ingestQueueRequest.ts:39-55` adds the partial unique live-target index, `server/src/ingest/requestQueue.ts:89-188` retries through duplicate-key races, and `server/src/test/unit/ingest-request-queue.test.ts:284-340` directly proves the concurrent first-submit collapse.  
  Proof status: direct.

- `waitForQueueRequestTerminalStatus()`  
  Candidate challenge: queue-state read failures could still break the blocking-wait contract even after the timeout-path cleanup fix.  
  Decision: endorsed finding.  
  Evidence: timeout-fallback rejection is covered at `server/src/test/unit/reingestService.test.ts:611-655`, but the initial read at `server/src/ingest/ingestJob.ts:2474` is outside that guarded path and the caller at `server/src/ingest/reingestService.ts:496-590` does not normalize arbitrary wait-helper failures.  
  Remaining weak area: no direct proof exists for initial `resolveQueueRequestRunState()` rejection before listener registration.  
  Proof status: missing for the challenged setup-read path.

- `recoverIngestQueueOnStartup()`  
  Candidate challenge: persisted `cleanup-blocked` state with a missing `runId` letting startup advance to newer work.  
  Decision: endorsed finding.  
  Evidence: the persisted schema and update helper both allow null `runId` values (`server/src/mongo/ingestQueueRequest.ts:20-35`, `server/src/ingest/requestQueue.ts:274-289`), but startup recovery only treats cleanup-blocked as blocking when `runId` is a non-empty string (`server/src/ingest/ingestJob.ts:2313-2327`). The existing proof at `server/src/test/unit/ingest-queue-runtime.test.ts:293-325` only covers the non-null `runId` case.  
  Remaining weak area: no direct proof covers malformed or partially written cleanup-blocked rows on restart.  
  Proof status: missing for the challenged mixed-shape persisted state.
