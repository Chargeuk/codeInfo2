## Plan Extraction

### Description Summary

Story 53 introduces an execution boundary for flows so a fresh run cannot accidentally reuse agent conversations from an older run. The implementation must persist `executionId`, keep slot reuse inside one execution only, preserve manual child-chat continuity on resume, and render lightweight `Run <shortExecutionId>` metadata without changing main conversation titles.

### Acceptance Criteria

1. Fresh flow starts create a new `executionId`.
2. The current flow execution persists `executionId` in `conversation.flags.flow`.
3. Fresh flow starts open a new parent flow conversation.
4. Fresh flow starts create a fresh per-execution agent mapping.
5. Same-slot agent reuse still works within one execution.
6. The persisted `agentConversations` mapping stays keyed by `${agentType}:${identifier}` and execution-scoped.
7. The runtime no longer relies on a process-global slot map across executions.
8. Re-running the same flow fresh creates fresh child agent conversations.
9. Resume continues the same `executionId`.
10. Resume continues the same child agent conversations for that execution.
11. Parent flow conversations persist `flags.flow.executionId`.
12. Flow-created child conversations persist `flags.flowChild.executionId`.
13. Child conversations remain visible in the Agents sidebar.
14. Repeated executions with identical titles show `Run <shortExecutionId>` metadata.
15. The run clue is derived from `executionId` without a separate ordinal.
16. The run clue appears in sidebar metadata, not titles.
17. Users can open child conversations from Agents and chat manually.
18. Resume continues the same child conversation after manual chat.
19. Manual child chat remains part of resumed context.
20. Resume uses the current persisted child conversation state, not a stale snapshot.
21. Child conversations remain normal agent conversations, not hidden flow-only resources.
22. Stopping preserves enough execution state to resume.
23. Fresh executions do not copy prior child slot mappings.
24. Legacy stopped flows without `executionId` remain resumable via backfill.
25. Concurrent fresh executions remain allowed across different parent conversations.
26. `RUN_IN_PROGRESS` remains scoped to one conversation.
27. Existing agent sidebar behavior stays intact apart from child visibility and run clue metadata.
28. Existing within-execution slot semantics remain intact.
29. Missing or wrong child mappings fail clearly.
30. Conflicting child execution markers fail clearly.
31. No broader multi-execution history subsystem is required.
