# External Review Findings - Story 0000047

## Findings

1. `must_fix` / `generic_engineering_issue` - `server/src/config/runtimeConfig.ts`

   The merge helper in `server/src/config/runtimeConfig.ts` currently normalizes any non-table `baseValue` or `runtimeValue` to `{}` inside `mergeNamedTables()` before `validateRuntimeConfig()` runs. That means malformed TOML such as `mcp_servers = "bad"`, `tools = "bad"`, or `projects = "bad"` can be silently dropped or replaced by inherited base values instead of producing the validation error the runtime-config path previously relied on. This is an endorsed external finding because it is a real merge-before-validate defect, it is not required by the story plan, and the current evidence artifact records this path as the highest-risk helper with missing direct proof for malformed top-level table rejection.

   Why it matters: this can hide user misconfiguration and allow execution to continue with unintended defaults, which is a correctness and runtime-safety issue rather than a cosmetic cleanup.

2. `optional_simplification` / `generic_engineering_issue` - `server/src/routes/chatProviders.ts`, `server/src/routes/chatModels.ts`, `server/src/mcp2/tools/codebaseQuestion.ts`

   The shared marker `DEV_0000047_T01_CODEX_DEFAULTS_APPLIED` uses raw Codex `model_source` values (`override|config|env|hardcoded`) in these emitters, while `server/src/routes/chatValidators.ts` logs normalized values (`request|config|env|fallback`). This is directionally valid feedback from the external review because the same marker now carries two vocabularies across surfaces, which makes downstream analytics and log parsing noisier than necessary. I am not endorsing it as a reopening correctness defect because Story 47 did not define a normalized observability schema for that field, raw Codex source data is still semantically useful, and the suggested automated patches shown in the external review would introduce duplicate object keys rather than a clean fix.

   Why it matters: this is an operational consistency cleanup opportunity, not a demonstrated product or contract bug.

## Rejected Or Non-Adopted External Review Comments

- Rejected: `server/src/routes/chatProviders.ts` comment claiming `resolveChatDefaults({}).warnings` are wrongly dropped from `/chat/providers`.

  Reason: Story 47 did not require a new response warning field here, and folding generic default-resolution warnings into `codexWarnings` would blur that field's meaning. The suggested automated patch only logs the warnings; it does not actually resolve the claimed response-contract issue. This is not strong enough to endorse as a finding.

- Rejected: `server/src/config/runtimeConfig.ts` comment suggesting `ensureChatRuntimeConfigBootstrapped()` should keep `stat.isFile()` and treat non-files as missing.

  Reason: the active plan explicitly says zero-byte files, unreadable files, invalid TOML files, and directories at the chat-config path count as existing rather than missing for Story 47. The current `fs.stat(...).then(() => true, ...)` behavior matches that edge-case contract. Reverting to `stat.isFile()` would conflict with the plan by making some existing paths eligible for bootstrap replacement.

- Partially valid but non-reopening: the three external comments about `model_source` inconsistency across `chatProviders`, `chatModels`, and `codebaseQuestion`.

  Reason: these are all the same underlying observability concern and are already captured above as one `optional_simplification`. They do not warrant three separate findings.

## Residual Risks And Weak-Proof Areas

- Payload-shape stability is still mostly indirect proof: route payloads appear unchanged and the client/manual/wrapper coverage remained green, but there is no dedicated contract snapshot artifact in this pass.
- The endorsed `mergeNamedTables()` finding is currently supported by code inspection and missing-negative-proof analysis rather than by an existing failing regression test in the branch.
