## Findings

1. `must_fix` `generic_engineering_issue` — `server/src/routes/conversations.ts:75`, `server/src/routes/conversations.ts:844`, `server/src/routes/conversations.ts:902`

   The new conversation working-folder edit route still normalizes missing or whitespace-only `workingFolder` input into a successful clear. The schema accepts `workingFolder` as optional and nullable, the route rewrites any absent or blank value to `undefined`, and the persistence call then converts `validatedWorkingFolder ?? null` into a clear operation. That means `{}` and `{ "workingFolder": "   " }` do not fail validation and instead silently erase the saved working folder. This is exactly the kind of invalid-input-to-success path the review pass was asked to catch, and it weakens the new API contract by making omission equivalent to an explicit clear even though the changed client helper in `client/src/api/conversations.ts` always sends either a trimmed string or `null`.

   Before/after contract comparison: the working-folder edit route is an additive contract surface and should stay backward compatible for valid callers, but the current omitted-field behavior is an unintended contract broadening that can clear persisted state without an explicit caller decision. The fix should require an explicit `null` to clear, keep a non-empty string for save, and reject absent or whitespace-only values as invalid request input. Direct proof is currently missing because there is no route-level test for omitted-field or blank-string payloads.

2. `should_fix` `generic_engineering_issue` — `server/src/flows/repositoryCandidateOrder.ts:110`, `server/src/flows/repositoryCandidateOrder.ts:121`

   `buildRepositoryCandidateLookupSummary()` still hides impossible caller mismatches by returning `fallbackUsed: false` when the selected repository is not present in the ordered candidate list. `findIndex()` returns `-1`, and the helper quietly reports the same fallback vocabulary it would use for a true first-choice selection. Today’s callers appear to pass only happy-path candidate winners, but this helper now feeds persisted `Turn.runtime.lookupSummary` data and shared resolution markers across multiple surfaces, so silently accepting a contradictory input would make downstream debugging ambiguous instead of surfacing an invariant break.

   Before/after contract comparison: the new lookup-summary fields and log markers are additive debug surfaces, but this helper can still emit a misleading schema/value combination under mismatched input. The helper should either throw or otherwise make the impossible state explicit, and it should gain a direct unit test for the contradictory-input path. The current protection is only indirect happy-path coverage in `server/src/test/unit/repositoryCandidateOrder.test.ts` and the higher-level integration suites.
