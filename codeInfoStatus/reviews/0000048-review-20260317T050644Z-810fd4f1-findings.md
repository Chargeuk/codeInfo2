## Findings

### 1. `must_fix` `plan_contract_issue`

Ingested-repository working-folder validation is silently disabled whenever repository enumeration fails, which lets the story fall back to “any existing absolute directory” instead of the plan’s canonical ingested-repository contract.

Files:

- [server/src/routes/chat.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/routes/chat.ts)
- [server/src/routes/conversations.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/routes/conversations.ts)
- [server/src/agents/service.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/agents/service.ts)
- [server/src/flows/service.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/flows/service.ts)
- [server/src/workingFolders/state.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/workingFolders/state.ts)

Why this is a finding:

- The route/service call sites catch repository-list failures to `undefined` or `[]`.
- [server/src/workingFolders/state.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/workingFolders/state.ts) treats missing or empty `knownRepositoryPaths` as “skip repository membership validation entirely.”
- That means a transient repo-enumeration failure changes the contract from “must be an ingested repository root” to “must be an existing absolute directory.”

Impact:

- `POST /chat` can accept a non-ingested absolute directory again during repository-list failure.
- conversation restore/list paths can preserve or restore non-ingested saved folders instead of clearing/rejecting them.
- agent/flow/direct-command paths can run against directories that are outside the ingested-repository set.

Why this violates the plan:

- Story 48 explicitly makes ingested-repository identity part of the canonical working-folder contract, and the review-fix Task 12 tightened chat to match that same rule.
- Silently bypassing the check on repository-list failure hides misconfiguration or runtime failure by defaulting too early.

Proof status:

- Direct code proof.
- Missing direct negative coverage for “repository enumeration failed, reject or surface operational failure instead of accepting.”

### 2. `must_fix` `plan_contract_issue`

The Task 15 runtime-config fix still silently defaults away a malformed top-level canonical runtime config container, so `window.__CODEINFO_CONFIG__` can be invalid without producing any diagnostic.

Files:

- [client/src/config/runtimeConfig.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/config/runtimeConfig.ts)

Why this is a finding:

- `readRuntimeConfig()` returns `{}` when `__CODEINFO_CONFIG__` is missing or is a non-object value, with no diagnostic.
- Arrays also slip through because `typeof [] === 'object'`, so a malformed array payload is treated as an object and all canonical fields read as `undefined`.
- The later field-level normalizers only emit diagnostics once a field is present but malformed. A malformed top-level canonical container never reaches that path.

Impact:

- A broken runtime injection can still look like a clean env/default fallback success.
- `DEV_0000048_T8_VITE_CODEINFO_RUNTIME_CONFIG` can incorrectly report `hasInvalidCanonicalConfig: false` for a malformed top-level canonical config payload.

Why this violates the plan:

- Task 15 explicitly exists to “Surface Malformed Canonical Client Runtime Config Instead Of Silently Defaulting It Away.”
- The field-level fix landed, but the top-level malformed-container case is still silently normalized into success.

Proof status:

- Direct code proof.
- Missing direct coverage for non-object and array-shaped `__CODEINFO_CONFIG__`.

### 3. `should_fix` `generic_engineering_issue`

Operational working-folder diagnostics are produced and then largely dropped: `causeCode` is preserved on the thrown error object, but the structured log omits it and some route-level error responses stringify the object to `[object Object]`.

Files:

- [server/src/workingFolders/state.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/workingFolders/state.ts)
- [server/src/routes/conversations.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/routes/conversations.ts)

Why this is a finding:

- `toUnavailableWorkingFolderError(...)` creates `WORKING_FOLDER_UNAVAILABLE` with a preserved `causeCode`.
- `restoreSavedWorkingFolder(...)` logs only `decisionReason: 'saved_value_unavailable'` and drops `causeCode`.
- Multiple route catches in [server/src/routes/conversations.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/routes/conversations.ts) serialize thrown plain objects with `` `${err}` ``, which yields `[object Object]` instead of the preserved reason/cause.

Impact:

- The review-fix split between stale and operational failures is correct in control flow, but the operational branch is not yet actionable enough for debugging.
- Production callers get a vague 500 payload and the shared working-folder marker loses the errno/cause detail that Task 13 intentionally added.

Why this matters:

- The story does not need to expose raw internals to end users, but it does need an actionable log trail for operational failures.
- This is a low-risk consistency cleanup in already-changed files and should be fixed rather than deferred.

Proof status:

- Direct code proof.
- Direct tests only prove the thrown object contains `causeCode`; they do not prove that callers surface it.

## Contract Comparison Notes

### API Routes

- `POST /chat` working-folder behavior is intentionally stricter than before Story 48 because the plan requires chat to enforce the same ingested-repository contract as agents and flows. This is intentionally breaking for previously accepted non-ingested absolute directories.
- Conversation working-folder save/restore endpoints in [server/src/routes/conversations.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/routes/conversations.ts) are additive and should be backward compatible for clients that do not use `flags.workingFolder`.
- The current review finding is not about the stricter contract itself; it is about silently disabling that contract on repository-list failure.

### Config Shapes

- Server env names changed to `CODEINFO_*` and client build/runtime names changed to `VITE_CODEINFO_*`.
- This cutover is intentionally breaking by plan; the plan explicitly rejected long-lived dual-read compatibility for repository-owned env names.
- The current runtime-config finding is not about that cutover being too strict. It is about malformed canonical runtime config being silently treated as absent.

### Persisted Artifacts

- `Conversation.flags.workingFolder`, flow resume-state working-folder fields, and `Turn.runtime.lookupSummary` are additive persisted-shape changes.
- These appear backward compatible for consumers that treat the new fields as optional.

### Wrapper Outputs

- `scripts/test-summary-server-unit.mjs` remains on the same summary-wrapper protocol surface.
- The review-fix work changed reliability, not the public wrapper protocol shape, so this appears backward compatible.

### Shared Log Marker And Event Schemas

- `DEV_0000048_T1_REPOSITORY_CANDIDATE_ORDER` looked schema-consistent across the inspected emitters in agents, flows, and markdown resolution.
- `DEV_0000048_T5_WORKING_FOLDER_ROUTE_DECISION` remains the main place where diagnostics are too lossy today because `saved_value_unavailable` does not carry the preserved failure detail.
- `DEV_0000048_T8_VITE_CODEINFO_RUNTIME_CONFIG` is internally consistent for field-level malformed values, but the top-level malformed-container case never reaches the diagnostic schema.
