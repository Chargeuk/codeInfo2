# Story 0000054 Findings

## Findings

1. `should_fix` `plan_contract_issue` `current_repository`

   The new dispatcher only enforces the waiting-queue cap if callers serialize `enqueue(...)` calls one at a time. Its capacity check looks at `queue.length` and `controllers.size` before the new item is pushed, but dispatch itself is deferred to a later microtask, so a same-turn burst of enqueues can overfill the queue before any request is counted as in-flight. That means the helper does not actually uphold the explicit bounded-queue contract on its own; it currently depends on the production caller’s sequential `await dispatcher.enqueue(...)` pattern in `processRun(...)`, and the changed test only proves the serialized case.

   Why this matters:
   - The plan’s queue-cap acceptance criteria describe a bounded dispatcher contract, not just a bounded outcome under one serialized producer.
   - The current proof is happy-path only: the changed unit test awaits each enqueue, so it never challenges a same-turn burst or a future multi-producer caller.
   - This is exactly the kind of concurrency-sensitive path that the findings pass is supposed to flag when correctness depends on an unstated serialization convention.

2. `should_fix` `generic_engineering_issue` `current_repository`

   `publishTerminalStatus()` computes terminal progress as `fileTotal / fileTotal`, which produces `NaN` when a re-embed finishes with `fileTotal === 0`. That state is reachable in the newly added AST-relevant deletions-only path: `workFiles` is empty, the run still rebuilds AST from remaining files, and the final status is published through the generic terminal path. In practice this serializes `percent` as `null`, so the terminal status payload is inconsistent exactly on the delete-only AST path that the new feature file added.

   Evidence:
   - [server/src/ingest/ingestJob.ts#L1299](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/server/src/ingest/ingestJob.ts#L1299)

   Why this matters:
   - The scenario is no longer theoretical; the changed feature explicitly exercises an AST-relevant deletion-only re-embed.
   - The branch does not currently prove terminal status shape for that path, so this slips through as a quiet payload inconsistency rather than a failing functional test.
   - This is a low-risk consistency bug in a file already changed by the story, so it should be cleaned up now instead of deferred.
