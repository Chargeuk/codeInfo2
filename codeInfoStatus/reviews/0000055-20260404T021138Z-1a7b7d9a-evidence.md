## Canonical Plan Extract

### Description

Story 55 adds a durable Mongo-backed whole-request queue for ingest and re-embed requests, keeps queue identity separate from runtime `runId`, dedupes by canonical target path, preserves blocking completion for flow/command/MCP callers, exposes queued visibility through the shared repository-list contract, and resumes queue work deterministically on startup.

### Acceptance Criteria

1. Ingest and re-embed requests use one durable MongoDB queue collection rather than being rejected immediately whenever another ingest run is active.
2. Every queueable request is normalized to the canonical embed target path before dedupe and queue insertion decisions are made.
3. If MongoDB is unavailable, queueable ingest and re-embed requests are rejected before any new run starts.
4. When MongoDB is unavailable, queueable ingest and re-embed requests return a clear retryable `QUEUE_UNAVAILABLE` error rather than a generic server failure.
5. The queue uses one collection for both waiting and currently running items; a running item remains in the collection until terminal completion.
6. Queue documents persist one coarse queue-state enum with exactly `waiting`, `running`, and `cleanup-blocked`.
7. If no ingest is active when a request is accepted, the server inserts the queue item and then starts that oldest queued item immediately.
8. If an ingest is already active when a request is accepted, the server inserts the queue item and leaves it waiting in FIFO order.
9. If a matching waiting queue item already exists, the server updates it in place, preserving queue identity and order metadata.
10. If a matching queue item is already running, the server still avoids creating a duplicate queue item, but it does not mutate the active run's settings mid-flight.
11. The duplicate-request rule applies across both start-ingest and re-embed requests once they resolve to the same canonical embed target path.
12. When a duplicate request is collapsed onto an existing queue item, the server returns success and reuses the existing queue request id instead of returning a duplicate-specific error.
13. Queue-aware success responses expose `queued`, `requestId`, and `queuePosition`.
14. `requestId` is the durable queue identifier for this story.
15. `runId` remains the active ingest-run identifier once work has actually started.
16. Queue-aware success responses do not expose a separate `deduped` field in this story.
17. For an immediately started request, the success payload includes the queue request id, the active `runId`, and a non-waiting queue state.
18. For a waiting request, the success payload includes `queued: true`, the queue request id, the current queue position, and no active `runId` yet.
19. `queuePosition` counts only waiting requests; an already running item uses the non-waiting state instead of reporting itself as queue position `1`.
20. Flow, command, and MCP re-embed callers that previously blocked until terminal completion still block until the queued request reaches a terminal outcome.
21. Queue wait time is treated as part of the blocking contract for those callers rather than as a reason to return early.
22. The preferred blocking path waits for real queue-item or ingest completion events rather than relying on a short fixed timeout as the normal control mechanism.
23. A much longer safety timeout may still exist as a final guard, but it is not the normal success path for correctly queued work.
24. The frontend ingest surfaces show when a repository is queued for re-embed and show its queue position in the repository list.
25. Queued state and queue position are added to the shared repository-list payload used by the ingest repo-list surfaces, rather than mirrored into unrelated flow-facing status payloads just for parity.
26. The shared repository-list schema version, REST `/ingest/roots` response, classic MCP `ListIngestedRepositories` response schema, and client normalization are updated together so queued rows can expose `requestId`, waiting-only `queuePosition`, and a `runId` that is absent until execution actually starts.
27. If a brand-new repository is queued before its first ingest run starts, the repository list still shows a temporary queued row right away.
28. Queued repository visibility is stable enough that a user can refresh or revisit the ingest page and still see that the request is waiting.
29. While a run is already active, the ingest UI still allows queueable start-ingest and re-embed submissions instead of blocking them behind a global active-run disable state; controls may remain disabled only when they are blocked for a separate destructive or invalid-state reason.
30. After a queueable submission is accepted, any later edits to the retained visible form fields affect only a later explicit submit; they do not silently mutate the already accepted queue item.
31. Removing the active-run disable gate does not change existing lock-driven field behavior: if the embedding model is locked, the model selector remains read-only and submissions keep using that locked model identity.
32. Existing single-flight ingest execution still applies at runtime: only one ingest or re-embed run is active at a time, even though multiple future requests may now be queued durably.
33. The queue request id may use the MongoDB document id generated for that queue item.
34. On startup, cleanup-blocked queue items are retried or resolved before the server starts any newer waiting item.
35. REST queue-backend outages return HTTP `503 Service Unavailable` with `QUEUE_UNAVAILABLE`; MCP, flow, and command surfaces preserve `QUEUE_UNAVAILABLE` as a structured retryable error instead of collapsing it into a generic invalid-request failure.
36. This story does not require queued-but-not-started requests to be removable by the user once they are in the queue.
37. Documentation and tests are updated to describe the durable queue contract, dedupe behavior, startup recovery, blocking caller behavior, and frontend queued-state visibility.

### Out Of Scope

- A separate `deduped` response field.
- A dedicated top-of-page queue card.
- Queue-position fields on unrelated non-repository payloads.
- Exactly-once crash semantics.
