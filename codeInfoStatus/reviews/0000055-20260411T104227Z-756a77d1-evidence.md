### Bulk Loop Outcome Semantics

- The current implementation intentionally keeps direct item action fetches in the helper loop, but it hoists non-item refresh work out of the loop so `onRefresh()` and `onRefreshModels?.()` only run after the batch settles and only when at least one item succeeded.
- Batch-level success and failure reporting is still able to distinguish full success from mixed success because `doReembed()` and `doRemove()` catch their own errors and return `{ ok, path }`; `handleBulk()` derives `failedPaths` and `successCount` from those normalized results rather than from thrown exceptions.
- The strongest direct proof for one-per-batch refresh cardinality currently exists on the bulk re-embed path. The bulk remove path has strong end-state proof for failed-row retention and mixed-success messaging, but refresh cardinality there is still inferred from the shared implementation rather than counted directly.

### Bulk Loop Hotspots For Findings
