## Findings

### 1. `should_fix` `generic_engineering_issue`

WebSocket conversation-summary upserts still normalize malformed present `source` and `flags` fields into success-shaped client state, even though the REST path was explicitly tightened to fail closed.

Files:

1. `client/src/hooks/useConversations.ts:101`
2. `client/src/hooks/useConversations.ts:117`
3. `client/src/hooks/useConversations.ts:508`
4. `client/src/api/conversations.ts:89`
5. `client/src/test/useConversations.source.test.ts:57`
6. `client/src/test/useConversations.source.test.ts:137`

Why this matters:

1. The REST list/update paths now validate conversation summaries with `parseConversationSummary(...)` and reject present invalid `source` or `flags` values.
2. The WebSocket path still goes through `normalizeConversationSummary(...)`, which coerces any non-`'MCP'` `source` to `'REST'` and feeds `flags` through `normalizeFlags(...)` without rejecting arrays or other malformed present values.
3. That means the same `ConversationSummary` contract is stricter on REST than on WebSocket. A malformed WS upsert such as `{ source: 'BROKEN' }` or `{ flags: ['not-an-object'] }` would still be treated as a valid conversation update instead of surfacing contract drift.
4. Story 48 depends on `flags.workingFolder` being trustworthy across restore/edit surfaces, so this is a real consistency issue in files already changed by the story and its review passes.

Before / after contract comparison:

1. Before Task 38, both surfaces were permissive and could normalize malformed fields into success-shaped state.
2. After Task 38, the REST path is intentionally stricter and fail-closed, but the WS path still preserves the old permissive behavior.
3. The resulting external contract is inconsistent rather than clearly backward compatible or clearly intentionally breaking.

Malformed-input question the findings pass asked explicitly:

1. If a WS upsert carries `source: 'BROKEN'`, `normalizeConversationSummary(...)` turns it into `'REST'` instead of rejecting it.
2. If a WS upsert carries `flags: ['not-an-object']`, `normalizeFlags(...)` spreads the array into an object instead of failing closed.
3. Current proof for those WS-specific bad paths is missing; the tests at `client/src/test/useConversations.source.test.ts` cover REST fetch failures and only happy-path WS upserts.

Suggested fix direction:

1. Route WS upserts through the same strict summary parser used by the REST path, or add equivalent strict validation for present `source` and `flags` before normalization.
2. Extend the `useConversations` tests with malformed WS upsert cases so the shared `ConversationSummary` contract is proven across both transport surfaces.
