## Focused Challenges

### 1. `createEmbeddingDispatcher`

- Contradictory input/state attempted:
  - `maxQueueSize=0`
  - same-turn enqueue burst before the pump runs
  - cancel landing after the first slot is occupied but before any late result is observed
- Evidence checked:
  - `server/src/ingest/embeddingDispatcher.ts`
  - `server/src/test/unit/ingest-dispatcher.test.ts`
  - `server/src/test/unit/ingest-cancel.test.ts`
  - `server/src/test/features/ingest-embedding-dispatch.feature`
  - `server/src/test/features/ingest-cancel.feature`
- Result:
  - strengthened rejected-risk conclusion
- Why:
  - The helper computes waiting work from immediate capacity, not raw queue length, so a zero waiting-queue cap still allows only immediately dispatchable work through.
  - The focused unit proof already covers same-turn burst admission/refill behavior, and the cancel proof already covers late-result ignore plus `waitForIdle()` completion after cancellation.
  - I did not find a narrower producer/consumer mismatch that would break the no-findings conclusion.

### 2. `chunkText` / `buildProseChunks`

- Contradictory input/state attempted:
  - a large Markdown file whose prose blocks barely overflow token limits, so a naive implementation would repeatedly retokenize the whole remaining tail or lose heading/fence boundaries while backing off
- Evidence checked:
  - `server/src/ingest/chunker.ts`
  - `server/src/test/unit/chunker.test.ts`
  - `e2e/ingest.spec.ts`
  - `planning/0000054-users-can-ingest-repositories-with-large-text-files-faster.md` Task 18 implementation notes
- Result:
  - left residual weak proof, but no new finding
- Why:
  - The local-cut path in `findLocalBoundaryCut(...)` and `sliceToFitLocally(...)` is clearly designed to avoid the older whole-tail fallback behavior.
  - The unit tests and browser/runtime proof confirm routing, token safety, and marker emission.
  - The remaining weakness is still the same one recorded in findings: there is no dedicated benchmark-style instrumentation artifact for the exact retokenization reduction under worst-case prose input. That is a proof-depth gap, not a concrete defect against the plan’s contract.
