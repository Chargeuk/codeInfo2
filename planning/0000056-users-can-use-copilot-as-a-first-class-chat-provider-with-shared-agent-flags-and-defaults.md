### Description

The product already has three chat providers in some form, but they do not feel equally supported from a user perspective. Codex currently has the strongest integration because it has repo-owned chat defaults, Codex-specific chat controls in the UI, and better alignment between the normal chat page and the MCP chat interface. GitHub Copilot support exists, but it is still a thinner integration. The page can select Copilot as a provider, yet Copilot does not currently receive the same tool wiring, the same default-config story, or the same provider-specific UI treatment that Codex receives.

This story closes that gap by making Copilot a first-class chat provider in the same product surfaces where Codex already feels strong, while also cleaning up the chat-provider architecture so it is no longer overly Codex-shaped. The user should be able to choose Copilot as the default chat provider, have the default chat model come from a provider-specific chat config file seeded by the app into the repo-local provider folder, use tools through the chat page and the chat MCP interface, and see a provider-neutral `Agent Flags` panel that shows only the options the selected provider actually supports.

The key design decision for this story is that the default-provider selector remains a single environment variable, but the default model no longer comes from a second shared environment variable. The environment variable `CODEINFO_CHAT_DEFAULT_PROVIDER` should remain the single top-level default selector. Once that provider is chosen, the server should load the selected provider's repo-owned chat config file and read the default model from there. This removes the need for `CODEINFO_CHAT_DEFAULT_MODEL` and avoids a confusing split where the provider comes from one place and the actual model comes from somewhere else.

The chosen config layout is intentionally symmetrical at the application layer:

- `codex/chat/config.toml` remains the repo-owned Codex chat-defaults file;
- `copilot/chat/config.toml` is added as a repo-owned Copilot chat-defaults file;
- `lmstudio/chat/config.toml` is added as a repo-owned LM Studio chat-defaults file.

These files are product-owned chat-default contracts seeded by the application into repo-local provider folders. They are not claims that all three providers natively read the same file shape themselves. Codex already has its own native runtime config under `codex/config.toml`. Copilot keeps its own auth and runtime home under `copilot/`. LM Studio keeps its own provider-specific runtime behavior. This story is about one consistent product-level defaults layer that the server reads and translates into each provider runtime, not about forcing every provider to adopt Codex's native on-disk contract.

The user has also now chosen that this product-owned defaults layer should stay normalized across providers instead of drifting into three unrelated local file shapes. In practice, that means `codex/chat/config.toml`, `copilot/chat/config.toml`, and `lmstudio/chat/config.toml` should all use one shared product-owned TOML format where provider-specific fields are added only when they are genuinely needed. Keeping the files in TOML preserves the current repo convention while still letting the server own one consistent parser, validator, and seeding path for all three providers.

That distinction matters most for Copilot. The user has chosen that Copilot auth and runtime state must stay under the existing `copilot/` home contract rather than moving into `copilot/chat/`. The new `copilot/chat/config.toml` file should therefore be treated as a repo-owned chat-defaults file only. It must not become Copilot's real runtime `configDir`, because doing that would risk separating chat defaults from the auth and state that the SDK and CLI already expect to live in the Copilot home.

The Copilot auth-home contract also now needs one explicit fragility repair. Live runtime evidence has shown that Copilot-managed files under that home can be rewritten in a JSON-with-comments style and that the current auth bootstrap path is too brittle because it still assumes one strict JSON artifact and one stale settings location. This story therefore now includes the in-scope work to make Copilot authentication persistence resilient to Copilot-managed JSONC artifacts, move the plaintext-persistence source of truth onto the live settings artifact the runtime actually uses, and stop blocking device auth just because a Copilot-managed compatibility file is commented or tool-owned.

The user has also now chosen that these provider folders should be seeded automatically rather than depending on a manually checked-in file for each provider. That means the local bootstrap path that already creates required repo-local folders for Codex and Copilot should be extended to create the LM Studio chat-defaults folder as well. The repo's gitignore and Docker build-context ignore rules should also keep the repo-local provider folders out of Git and out of image build inputs, including the new `lmstudio/` folder, so runtime defaults and local provider state do not create accidental source-control noise or get copied into Docker builds.

The remaining LM Studio folder decision is also now fixed. In this story, the new `lmstudio/` folder is only an app-managed seeded chat-defaults folder. It is not a new LM Studio runtime-state home. That means this story can seed `lmstudio/chat/config.toml` and keep it out of Git without also taking on a larger contract around LM Studio-native state, LM Studio-native MCP files, or a new Docker-mounted provider home for LM Studio itself. If the product later wants LM Studio-native runtime files in a repo-local home, that should be treated as a follow-up story rather than being silently folded into this one.

The same principle applies to tool support. Research and repo evidence show that Copilot can support tools through the SDK session layer, but the current repository does not actually wire that path up in normal chat execution. This story should therefore make app-managed tool parity the primary mechanism. In practice that means the server should construct Copilot tools from the repository's own trusted tool layer and pass them into Copilot session creation, rather than depending on provider-native MCP file discovery as the main product contract. Native provider-specific MCP config may still exist, but it is not the primary parity mechanism for this story.

The chat-page user experience also needs to become provider-neutral. Today the page has a `Codex flags` panel, uses Codex-specific defaults, and sends a `codexFlags` payload only for Codex. That architecture is too narrow now that the product supports Codex, Copilot, and LM Studio. This story should replace the Codex-only control path with a provider-neutral `Agent Flags` contract. The selected provider should advertise which flags or provider-specific options are supported, what their defaults are, and whether they are editable. The client should then render only those controls. This means Codex may still show the richest set of editable controls, while Copilot or LM Studio may show fewer. That is acceptable. The important rule is that the UI must not invent fake controls for providers that do not truly support them.

This story also now includes a dependency-refresh pass for the chat providers that most affect that contract. The current repository is behind on the published Codex package versions, so implementation should update `@openai/codex` and `@openai/codex-sdk` to the latest published versions verified at implementation time. The GitHub Copilot SDK version should also be rechecked during implementation so the story can move to the latest published Copilot API if a newer package is available by then. The LM Studio SDK version should likewise be rechecked during implementation, but because it was already current when this requirement was added, the story only needs to bump `@lmstudio/sdk` if a newer release exists at implementation time rather than forcing churn for its own sake. The reason that dependency refresh belongs in this story is that newer Codex releases expose a broader config surface than the app currently models, and the product also wants the Copilot integration reviewed against the latest published SDK contract rather than against a stale assumption.

The user has also chosen the preferred shape of that provider-neutral `Agent Flags` contract. Each provider implementation should expose a common option-descriptor method that returns a JSON array of the provider's exposed options. Each array item should represent one exposed option and include at least: a stable option key, a human-facing label, the UI control type the client should render, the supported values or value bounds, a hardcoded seed default used when the provider chat config is first created, and the resolved default currently loaded from that provider's `chat/config.toml`. The contract should preserve both the original seed default and the currently resolved config-backed default instead of collapsing them into one field, so bootstrap behavior and later operator edits stay visible and testable.

That option-descriptor contract should also be model-aware rather than provider-only. Some options, such as Codex and Copilot reasoning effort, can depend on the selected model instead of only the provider. Because of that, the shared provider option method should accept the selected model when resolving the available options and their effective defaults. The user has now chosen that the product should not fetch those flags through a second independent API just for flags. Instead, the server should return one combined models-and-flags payload where common provider flags are returned once and model-specific differences are returned as per-model overrides or capability narrowing inside the same response. In practice, the client should refresh that combined payload when the user changes provider, and use the selected model's override data when the user changes model, so the UI always reflects the actual provider-plus-model contract without splitting models and flags across separate fetches.

The client should use that server-provided option-descriptor JSON as the source of truth for building the `Agent Flags` controls rather than keeping provider-specific UI branching in the page. The initial values shown to the user should come from the resolved defaults loaded from the provider's current `chat/config.toml`, not directly from the hardcoded seed defaults, because the local config may already have been edited. Once the user changes those options in the UI, the selected values should be sent with each normal chat request so the request contract remains explicit and stateless at send time even though the defaults were seeded from provider-local config. The MCP chat interface remains out of scope for this flags payload in this story and should continue to rely on provider and model defaults only.

The user has also now chosen the first supported editable option sets for the new providers. Copilot should not pretend to have the exact Codex flag set. Instead, its first supported editable options should be the real Copilot-aligned options the repository can honestly support now: reasoning effort and one tool-access setting. That tool-access setting should be a product-level abstraction that controls the built-in Copilot tool and MCP surface as a whole, even though the SDK maps it through allowlist or denylist fields rather than through one native enum. The first visible Copilot `Agent Flags` should therefore stay intentionally smaller than the raw SDK surface. `workingDirectory` should stay out of scope for this story because the SDK already falls back to the current process working directory when no override is supplied, and the product does not currently need a second UI control for that behavior. Copilot permission behavior should also stay out of the visible `Agent Flags` surface for this story. The product wants Copilot to run without interruption in the containerized environment, so the first implementation should treat permissive non-interrupting approval as an implementation/defaults decision by using the SDK's `approveAll`-style permission path or an equivalent always-approve handler. Fine-grained Copilot approval controls and per-server MCP toggles can be follow-up stories if the product later wants them.

The user has also now chosen the exact first-pass shape of that single Copilot tool-access control. It should start with just `On` and `Off`, not a middle mode. `On` should map to the normal allowed Copilot tool surface for this product, and `Off` should map to no tool access. That keeps the first Copilot UI very easy to understand while still leaving room for a future story to add a safer middle mode if the product later wants one.

LM Studio should also get real editable options in this story, but they should be LM Studio-native generation or tool options rather than copied Codex labels. The initial LM Studio set should therefore be built around native generation or tool controls such as temperature, max tokens, context overflow handling, and tool access. This story should not add fake LM Studio controls that merely look like Codex settings without matching LM Studio behavior underneath.

The Codex side of that option contract is also now more precise than the current UI. The first provider-neutral Codex option set should include sandbox mode, approval policy, reasoning effort, reasoning summary, verbosity, network access, and web search mode. `modelReasoningSummary` and `modelVerbosity` are newly in scope because current Codex releases expose them as simple, user-understandable dropdowns, and they fit naturally into the new option-descriptor contract. The old boolean `webSearchEnabled` should be replaced by a real `webSearchMode` option so the product can express `disabled`, `cached`, and `live` instead of collapsing everything into on or off. The seeded Codex chat config should use the most permissive supported default for that field, which means `live`.

Approval behavior also needs an explicit simplification boundary. The old flat `on-failure` choice should be removed from the supported editable Codex UI for this story, and the first supported Codex approval choices should be limited to the remaining simple contract the user wants to keep. Newer Codex releases support a richer granular approval structure, but that is intentionally out of scope here because it would turn one dropdown into a larger multi-field permissions design. The story should therefore refresh Codex to the newer library versions without taking on the full granular-approval UI at the same time.

The same “simple where honest” rule applies to Codex network access. Newer Codex config expresses this as `sandbox_workspace_write.network_access`, which is still just a boolean. That means the product can keep the simple `networkAccessEnabled` flag in this story as long as it is documented and implemented as the workspace-write network-access toggle rather than as a provider-wide permission master switch. The seeded Codex default for that flag should be permissive and default to `true`.

This newer Codex contract also has a migration implication for existing local repos. Older local `codex/chat/config.toml` files may still contain removed values such as the old `on-failure` approval mode or the old boolean-style web-search setting. The user has chosen that this story should handle those older local configs by normalizing them forward and warning clearly, rather than by failing chat until the operator edits the local file by hand.

Changing Agent Flags should keep the user in the same conversation rather than automatically starting a fresh conversation. This matches the current Codex behavior in the chat page, where provider and model changes create a new conversation for the next send but flag changes do not. For this story, provider-neutral Agent Flags should preserve that existing interaction model: a flag change affects later sends in the current conversation instead of resetting the visible transcript.

Provider and model changes need equally explicit state rules. In this story they remain next-send mode switches rather than same-conversation mutations: when the user changes provider or model, the active Agent Flags draft should be rebuilt from the newly selected provider/model contract, any now-incompatible hidden flags should be cleared from the draft immediately, and the next send must not reuse the prior conversation's provider-specific persisted state such as Codex `threadId`. If the user selects an existing conversation from history after staging a provider/model change, the restored conversation state wins and the unsent next-send draft must not bleed into that restored conversation.

This story also extends the same parity principle to the MCP chat interface. The current `codebase_question` MCP tool still hardcodes provider support to Codex and LM Studio. Users want Copilot to feel equally usable when chat is driven through the MCP surface, not only when it is driven through the browser page. This story therefore includes provider-contract work so Copilot can be selected through the chat MCP interface anywhere that the current chat provider contract is meant to be shared.

Another important contract decision is that fallback behavior remains provider-first. If `CODEINFO_CHAT_DEFAULT_PROVIDER` points to a provider that is unavailable or misconfigured, the server should fall back to the next available provider using that fallback provider's own `chat/config.toml` default model. This story should not preserve a hidden hardcoded or env-driven default-model back door once provider-local config files become the source of truth. If a provider-specific chat config file is missing, unreadable, or invalid, that should be treated as a provider misconfiguration that can trigger warning logs and provider fallback rather than silently selecting a surprising model from somewhere else.

That fallback rule is now more specific for explicit user choice too. Automatic fallback is acceptable when the system is resolving the default provider and the user did not explicitly choose one for the current request. But if the user explicitly picks Codex, Copilot, or LM Studio and that provider's local config is broken, the product should show an error rather than silently switching to a different provider behind the user's back. That keeps provider choice trustworthy and avoids confusing “I selected Copilot but got Codex” behavior.

The remaining config decision is also now fixed. Each provider's `chat/config.toml` should store both the default model and that provider's default Agent Flags where the repository actually supports them. This makes the provider-local chat config the single source of truth for supported chat defaults instead of repeating the current Codex split where some defaults come from `codex/chat/config.toml`, some come from env vars, and some are still hardcoded. This story therefore includes a small Codex cleanup as well as new Copilot and LM Studio config seeding.

Provider-local chat config changes should be reflected by the next relevant request rather than only after a restart. This matches the current Codex precedent, where defaults are reread through normal request-time resolution instead of a separate file watcher. In practice, that means a user who edits `chat/config.toml` should see the new default behavior on the next matching chat or model/default lookup request without needing a server restart, while the story stays out of scope for background live-watch hot reload.

This story is intentionally focused on normal chat and chat-MCP parity. It should not expand into every other agentic execution surface in one go. General agent runs, flows, and command execution already have their own provider and tooling assumptions and should only be touched where they are required for the chat MCP parity contract being targeted here.

Overall, when this story is complete:

- a user can choose Copilot as the default chat provider through the existing provider-default env path;
- the actual default Copilot model comes from `copilot/chat/config.toml`;
- the chat page shows `Agent Flags` rather than `Codex flags`;
- each provider shows only its supported options;
- Copilot can use tools through the normal chat page and the shared chat MCP interface;
- provider-local chat defaults feel consistent across Codex, Copilot, and LM Studio;
- Copilot auth and runtime state remain safely rooted in `copilot/`;
- valid provider-local chat config files are seeded automatically for Codex, Copilot, and LM Studio in repo-local folders that stay out of Git and out of Docker build context noise.

## Edge Cases And Failure Modes

- If `CODEINFO_CHAT_DEFAULT_PROVIDER` resolves to a provider whose provider-local `chat/config.toml` is missing, unreadable, or invalid, the default-provider path must fall back to the next available provider using that fallback provider's own config-backed default model. If the user explicitly selected the broken provider for the current request, the request must fail clearly instead of silently switching providers.
- Older local Codex chat defaults may still use removed values such as `on-failure` or a boolean-style web-search setting. The story must normalize those values forward with a warning on the normal read path, and later proof must distinguish successful normalization from hard failure so local repositories do not break simply because their seeded file is older than the new contract.
- Provider and model switching already change the visible chat state in [client/src/pages/ChatPage.tsx](client/src/pages/ChatPage.tsx) and [client/src/hooks/useChatModel.ts](client/src/hooks/useChatModel.ts), while existing flag changes stay in the same conversation. The new provider-neutral Agent Flags path must therefore keep that boundary explicit: provider or model changes rebuild the active flag draft for the new selection, clear incompatible hidden values immediately, and prevent old hidden values from being submitted or persisted under the new selection.
- For this story, hidden or no-longer-supported Agent Flags should not linger as active draft state after a provider or model change. The client should rebuild the visible flag draft from the newly selected provider/model contract and clear incompatible values immediately, and the server should still reject any contradictory stale payload that leaks through.
- If the user stages a next-send provider/model change and then restores a saved conversation, the restored conversation state wins. The app must not merge the unsent draft provider/model or hidden Agent Flags into the restored conversation, and later proof should distinguish "restored conversation won" from "stale draft was merely omitted on send."
- `networkAccessEnabled` remains meaningful only for Codex `workspace-write` sandbox behavior. The provider-neutral UI and server contract must make that scope explicit so Copilot and LM Studio do not receive a fake network toggle, and Codex requests outside `workspace-write` do not pretend the setting is a provider-wide permission switch.
- The first Copilot tool-access setting must prove both `On` and `Off` behavior through the normal session-creation path in [server/src/chat/interfaces/ChatInterfaceCopilot.ts](server/src/chat/interfaces/ChatInterfaceCopilot.ts). `Off` must remove tool access deterministically, `On` must restore the repository-managed tool surface, and neither path should require per-server MCP toggles or repeated approval prompts to behave correctly.
- Codex compatibility now has two separate contracts that the story must keep distinct. The installed `@openai/codex-sdk` thread options still accept legacy values such as `approvalPolicy = on-failure` and `webSearchEnabled`, while the newer product-facing direction for this story is `webSearchMode` plus a smaller editable approval set. That means the read path must continue to accept and normalize legacy Codex disk/env values, but newly seeded config comments, outward-facing Agent Flags, and editable defaults must use the newer product vocabulary instead of writing new legacy keys back out.
