## Findings

### 1. `must_fix` `plan_contract_issue`

The shared scroll-anchor logic moves the reader whenever total transcript height grows while they are scrolled away from the bottom, even if the height change happened below the viewport instead of above it. In [SharedTranscript.tsx](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/components/chat/SharedTranscript.tsx#L200) the `reconcileScrollPosition` path computes only `deltaScrollHeight`, and when the mode is `scrolled-away` it always adds that delta to `scrollTop` at [SharedTranscript.tsx](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/components/chat/SharedTranscript.tsx#L233). That preserves distance from the bottom, not the reader’s visible anchor. If new content or row growth happens below the current viewport, this code will still push the reader downward, which is the opposite of the plan’s “manual-scroll-away should preserve reading position” contract.

Why this matters:

- The plan explicitly requires that when the user has scrolled upward, new content must not force a jump and visible reading position must remain stable.
- The current logic cannot distinguish “growth above me” from “growth below me,” so it is structurally over-broad.

Proof status:

- Current tests give only indirect coverage for this risky path. The scroll tests exercise anchor preservation for above-viewport or within-viewport growth, but I did not find direct proof for “height grows below the viewport and `scrollTop` must remain unchanged.”

What findings should inspect next:

- whether the resize signal can be narrowed to rows above the viewport;
- whether anchor preservation should depend on the changed row’s position rather than total `scrollHeight` delta alone;
- whether the same over-adjustment exists in the virtualized row-growth path.

### 2. `should_fix` `generic_engineering_issue`

Malformed `USE_BROWSER_HOST:<port>` runtime config now silently degrades into a successful browser-origin API base URL instead of surfacing a hard configuration failure. The new directive parsing in [apiBaseUrl.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/config/apiBaseUrl.ts#L18) and [runtimeConfig.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/config/runtimeConfig.ts#L228) returns a diagnostic when the directive is malformed, but `resolveClientRuntimeConfig` still falls back to `getBrowserOrigin()` at [runtimeConfig.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/config/runtimeConfig.ts#L325). Callers of `getApiBaseUrl()` only receive the fallback string, and outside tests the diagnostic is only emitted in a one-time console info log at [runtimeConfig.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/config/runtimeConfig.ts#L287). That hides a bad explicit config behind a seemingly valid URL and can redirect all API traffic to the wrong origin.

Why this matters:

- This review specifically asked for fallback-selection and malformed-input checks.
- The new config shape is additive, but the malformed-directive behavior is not safely self-revealing.
- The change touches a shared config surface used by many hooks and API helpers, so a masked misconfiguration has wide blast radius.

Before/after contract comparison:

- Before: `apiBaseUrl` accepted literal URLs and otherwise fell back to browser origin.
- After: it additionally accepts `USE_BROWSER_HOST:<port>`.
- Backward-compatibility status: additive for well-formed inputs, but unclear and risky for malformed directive inputs because explicit user intent is replaced by fallback success rather than rejected or surfaced strongly.

Proof status:

- Direct proof exists, but it proves the problematic behavior: [baseUrl.env.test.ts](/Users/danielstapleton/Documents/dev/codeinfo2/codeInfo2/client/src/test/baseUrl.env.test.ts#L167) asserts that malformed directives fall back to `window.location.origin`.

What findings should inspect next:

- whether malformed explicit directives should block startup or at least surface an actionable runtime warning outside console-only diagnostics;
- whether callers that currently only use `getApiBaseUrl()` need access to invalid-config state;
- whether runtime config precedence should refuse fallback when an explicit directive was supplied but invalid.

Repo search in this review pass found no non-logger callers of that global hook; it exists only to support Story 49’s manual closeout markers. This is not a correctness failure, but it is more global and permanent than the story appears to need.

Why this matters:

- It adds a persistent story-specific browser API rather than reusing the existing generic manual-acceptance hook shape.
- It slightly increases long-term maintenance burden in a shared logger module.

Before/after contract comparison:

- Before: generic manual acceptance globals only.
- After: generic globals plus story-specific `__story0000049ManualValidation`.
- Backward-compatibility status: additive and backward compatible, but broader than necessary.

Proof status:

- Direct proof exists that the hook is installed, because it is wired directly in the logger module.
- I found no direct proof that this story-specific global must remain after the story closes.
