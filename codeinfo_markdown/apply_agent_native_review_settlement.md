# Apply the agent-native review settlement

Read the complete-pass settlement recommendation, every referenced batch, factual record under the active pass's `attempts/` directory, every supported finding, current Git state, and the bounded plan. Interpret self-describing evidence by meaning and tolerate understandable imperfections; do not require exact filenames, schemas, reviewer counts, or scheduling-group identities. A failed launch without a batch remains honest unavailable coverage and must not disappear from closeout.

Read `$CODEINFO_ROOT/codeinfo_markdown/task_up/01-shared-contract.md` and `$CODEINFO_ROOT/codeinfo_markdown/task_up/09-proof-and-testing.md` before changing the plan. In every task you create or update, `Testing` contains only runnable automated proof. Preserve optional browser, screenshot, agent-driven, or other manual scenarios only in checkbox-free, non-blocking `Manual Testing Guidance`. Never create an unchecked manual-proof checkbox or a blocker whose only owner is the later manual-testing agent.

If the pass was genuinely clean, record the no-work closeout without adding a task. If either repair agent committed fixes and nothing remains after both repair opportunities, add or update one final testing/revalidation task after all existing work. If findings remain after the stronger repair opportunity, create one or more appropriately scoped implementation tasks, improve their dependencies, exit criteria, subtasks, automated proof, manual guidance, target-repository ownership, and finding provenance, then add one final testing/revalidation task after them. Do not task a disposition prediction or a finding already fixed on a later target HEAD.

Use the repository plan helpers repeatedly to check the work. Before returning, semantically inspect every task you created or changed and repair any manual instruction accidentally placed in `Subtasks` or `Testing` by moving or merging its meaning into `Manual Testing Guidance` and removing only the misplaced checklist item. Keep task ordering and current-task handoffs valid. Update the plan continuously and make the normal implementation loop able to select the new work. Do not implement remaining findings or start another review in this step.
