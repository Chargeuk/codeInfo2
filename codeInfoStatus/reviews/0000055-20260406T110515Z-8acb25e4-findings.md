# Review Findings

## Findings

1. `must_fix` `plan_contract_issue` `current_repository`
   - Queued re-embed admission now rewrites the selected repository root to the mounted workdir path before execution, but the runtime still keys delta lookup and cleanup by the execution-time `root`. In [server/src/ingest/reingestService.ts#L401](/home/d_a_s/code/codeInfo2/server/src/ingest/reingestService.ts#L401), `buildQueuedReingestRequest()` stores `resolvedRunPath` as both `canonicalTargetPath` and `requestPayload.path`. That value becomes the actual run input in [server/src/ingest/ingestJob.ts#L394](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts#L394), and `processRun()` then performs re-embed delta lookup and cleanup against that discovered `root` in [server/src/ingest/ingestJob.ts#L898](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts#L898) and [server/src/ingest/ingestJob.ts#L1472](/home/d_a_s/code/codeInfo2/server/src/ingest/ingestJob.ts#L1472). For repositories still indexed under legacy `/data/...` roots, the queued re-embed can miss existing ingest-file metadata and stale vector/root cleanup, effectively treating the run as a fresh root under `/home/...` instead of re-embedding the existing repository. The new proof only asserts that the remap happens in [server/src/test/unit/reingestService.test.ts#L521](/home/d_a_s/code/codeInfo2/server/src/test/unit/reingestService.test.ts#L521); it does not prove that the original persisted root is still reused or dual-cleaned.

2. `should_fix` `plan_contract_issue` `current_repository`
   - The repository-list source of truth can now emit duplicate rows for one path after repeated ingests, because the current live path no longer dedupes persisted root metadata before applying queue/active overlays. [server/src/lmstudio/toolService.ts#L875](/home/d_a_s/code/codeInfo2/server/src/lmstudio/toolService.ts#L875) builds `repos` from every metadata row, `repoBySourceId` at [server/src/lmstudio/toolService.ts#L951](/home/d_a_s/code/codeInfo2/server/src/lmstudio/toolService.ts#L951) only keeps one winner for later overlays, and [server/src/routes/ingestRoots.ts#L163](/home/d_a_s/code/codeInfo2/server/src/routes/ingestRoots.ts#L163) now returns `payload.repos` directly. That means queue or active overlays update only the row that won the map entry while stale duplicates remain in the response, which conflicts with the Story 55 requirement that the shared repository list be the stable queued-state source of truth. The retained dedupe proof is stale: the helper in [server/src/routes/ingestRoots.ts#L119](/home/d_a_s/code/codeInfo2/server/src/routes/ingestRoots.ts#L119) still has tests in [server/src/test/unit/ingest-roots-dedupe.test.ts#L58](/home/d_a_s/code/codeInfo2/server/src/test/unit/ingest-roots-dedupe.test.ts#L58), but that helper is no longer on the production response path.

3. `should_fix` `generic_engineering_issue` `current_repository`
   - The row-level re-embed UI accepts any `2xx` payload as success and no longer verifies the queue-aware acceptance contract that this story introduced. In [client/src/components/ingest/RootsTable.tsx#L160](/home/d_a_s/code/codeInfo2/client/src/components/ingest/RootsTable.tsx#L160), `doReembed()` parses `queued`, `requestId`, `runId`, and `queuePosition` but never checks that `requestId` exists or that the response contains either `queued: true` or a non-empty `runId`; it will still show "Re-embed started" for a malformed `202`. The same story already validates `requestId` in [client/src/components/ingest/IngestForm.tsx#L208](/home/d_a_s/code/codeInfo2/client/src/components/ingest/IngestForm.tsx#L208), so the client-side contract is now inconsistent across the two changed ingest entry points. The current tests only cover valid row-level responses in [client/src/test/ingestRoots.test.tsx#L121](/home/d_a_s/code/codeInfo2/client/src/test/ingestRoots.test.tsx#L121), so this malformed-success case is unproved.
