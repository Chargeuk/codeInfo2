# Story 0000043 PR Summary

## What Changed

- Server websocket stop handling now accepts conversation-only `cancel_inflight` during the startup race, keeps explicit invalid-target failures for stale `{ conversationId, inflightId }` requests, and emits non-terminal `cancel_ack { result: 'noop' }` only for the no-active-run path.
- Runtime ownership is now server-authoritative: the conversation lock stores active `runToken` ownership, the inflight registry stores token-bound pending-cancel state, and chat, agent instruction, command, and flow execution all clear inflight state, pending cancel, and lock ownership through one cleanup path.
- Shared client stop handling is now split cleanly between `useChatWs` and `useChatStream`: the websocket hook sends conversation-only stop when `inflightId` is not yet known and surfaces `cancel_ack`, while the stream hook owns `stopping`, no-op recovery, and final `stopped` reconciliation without inventing a local terminal bubble.
- Chat, Agents, and Flows page UX now match the same stop contract: visible `Stopping` while cancellation is pending, visible `Stopped` only after `turn_final.status === 'stopped'`, persisted stopped hydration after reload, same-conversation reuse after confirmed stop, and exact `[stop-debug]` browser markers for manual verification.

## Stop Contract Reviewers Should Expect

- Pages always send `cancelInflight(conversationId, inflightId?)`.
- Startup-race stop is valid with `conversationId` only when the page does not yet know the current server-visible `inflightId`.
- `cancel_ack.result === 'noop'` is request-correlated, non-terminal, and only clears `stopping` for the no-active-run path.
- A real successful stop is confirmed only by `turn_final.status === 'stopped'`.
- Same-conversation reuse must work immediately after cleanup completes, and stale stop requests must not cancel a later replacement run.
